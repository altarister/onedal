# 🗺️ 1DAL 풀 스택 라이프사이클 맵핑 리포트 작성 가이드

이 문서는 1DAL의 안드로이드 앱, 서버(Node.js), 프론트엔드(React)에서 각각 추출한 로그를 바탕으로 `1DAL_Lifecycle_Report.md` 를 작성하는 **표준 방법론**을 정의합니다.

모든 리포트는 **반드시 `ROADMAP_sequenceDiagram.md`의 생명주기를 완벽하게 따라야 하며, 시간대별로 누락되는 로그가 없도록** 꼼꼼하게 교차 검증하여 작성해야 합니다.

---

## 📌 핵심 원칙 (Rules)

1. **시간대(Timestamp) 정렬 및 동기화**
   - 3개의 파일(`app.txt`, `server.txt`, `front.txt`)에 기록된 시간을 기준으로 사건을 오름차순 정렬합니다.
   - 단, 기기(OS 로컬시간, 예: 18:00 KST)와 서버(UTC 기반, 예: 09:00 UTC)의 시간이 다를 경우 이 시차를 보정하여 같은 시간대의 사건으로 묶어줍니다.
   
2. **단 1줄의 `[ROADMAP]` 로그도 누락 금지**
   - 불필요해 보이는 로그라도 생명주기에 포함되는 디버깅 지표이므로, `[ROADMAP]` 태그가 붙은 모든 플랫폼의 로그를 시간 순서대로 빠짐없이 기입합니다.
   
3. **시퀀스 다이어그램 기반의 명확한 단계 분기**
   - `ROADMAP_sequenceDiagram.md`에 명시된 플로우에 따라, 무조건 아래의 **[STEP 1] ~ [STEP 8]** 단계로 섹션을 분기하여 그룹핑합니다.

4. **통신 페이로드의 원본 스키마(Schema) 및 타입(Type) 명시**
   - 소켓(Socket)이나 HTTP 통신을 통해 서버, 기기, 관제웹 간에 실제 교환되는 핵심 데이터(예: 콜 상세 객체, 통신 상태, 필터값 등)는 임의로 축약하지 않습니다.
   - 설계된 데이터 스키마와 타입(Type)에 의거하여 전달된 모든 키워드와 값을 로그 내에 충실히 표시함으로써, 데이터 송수신 과정에서의 누락이나 에러 여부를 확실히 추적할 수 있게 기록합니다.

5. **프론트엔드 로그 파일명 전처리 (Cleaning)**
   - 브라우저 콘솔 로그 특성상 맨 앞에 따라붙는 파일명 및 라인 번호(예: `PinnedRoute.tsx:65 ` 또는 `installHook.js:1 ` 등)는 가독성을 해치므로, 스크립트 단에서 정규식을 활용해 깨끗하게 지우고 순수 메시지만 리포트에 기록해야 합니다.

## 🗂️ 다이어그램 기준 단계별 로깅 맵핑 방법

### 🟢 [STEP 1] 관제탑 서버 기동 및 필터 동기화
- **정의**: 서버가 구동되고, 관제웹이 연결되며, 안드로이드 기기가 소켓/폴링에 합류하는 초기 세팅 구간입니다.
- **포함 대상 로그**:
  - `서버 기동 및 디폴트 필터 셋업`
  - 안드로이드 기기의 `[HTTP 폴링] POST /api/scrap` 및 `[필터 동기화 적용됨]`
  - 관제웹의 접속 및 필터 정보 렌더링

### 🟢 [STEP 2] 첫짐 1차 선점 (단독콜 사냥)
- **정의**: 기사 화면에 꿀콜이 등장하고 앱의 `AccessibilityService`가 텍스트를 감지하여 광클하는 단계입니다.
- **포함 대상 로그**:
  - `화면 변경 감지 | 화면: LIST | 모드: AUTO`
  - `타겟 콜 파싱 결과 ... ✅ 🎯 [5대 조건 통과!]`
  - `✅ [가로채기 성공!] 화면 좌표 터치 완료! (리스트 클릭)`

### 🟡 [STEP 3] 1차 확정 통신
- **정의**: 인성 콜 상세 화면에 진입하자마자 **'확정'** 버튼을 뉴럴 클릭하고, 이 사실을 서버와 관제탑에 최우선 통보하는 단계입니다.
- **포함 대상 로그**:
  - 앱의 `확정 버튼 광클 (배차 시도)`
  - 서버의 `POST /orders/confirm 확정정보 수신` 및 `대기 필터로 설정값 업데이트 (isActive: false)`
  - 관제웹의 `order-evaluating | 전표 → 신규` 수신

### 🟢 [STEP 4] 2차 상세 수집: 팝업 자동 서핑
- **정의**: '확정' 버튼 격발 성공 후, 배차된 콜의 정확한 정보(적요상세, 출발지, 도착지)를 얻기 위해 자동으로 팝업을 열고 닫는 무인 서핑 단계입니다.
- **포함 대상 로그**:
  - `🏄‍♂️ [자동 팝업 서핑] 확정 화면 진입 확인! 적요상세 팝업 호출 시도`
  - 적요상세, 출발지, 도착지 각각의 `📝 스크래핑 성공! 닫기 버튼 누름`
  - 최종 `POST /orders/detail 상하차지 + 적요내용 정보 수신`
  - 관제웹의 `order-detail-received` 수신

### 🟢 [STEP 5] 카카오 연산 3중 폴백 & 자동 회랑 전개
- **정의**: 수집된 텍스트 주소를 카카오 API로 좌표로 변환하고 실제 거리 및 시간을 예측하는 서버와 관제탑의 연산 구간입니다.
- **포함 대상 로그**:
  - 서버의 `🛡️ [카카오 API 3중 폴백] 괄호제거 ➡️ 주소검색...`
  - 서버 및 관제웹의 `🔎 [카카오 연산 완료] 단독 xx km, xx분 '똥'` 도출 로그

### 🟢 [STEP 6] 관제탑 결재 (취소 vs 유지)
- **정의**: 관제원이 추천 결과를 바탕으로 수락(Keep) 혹은 거절(Cancel)을 누르는 판단 로직입니다.
- **포함 대상 로그**:
  - 프론트엔드 버튼 클릭 `👆 프론트에서 '유지 확정(KEEP)' 버튼 클릭`
  - 서버의 `합짐 필터(isSharedMode: true)로 설정값 업데이트 (합짐 사냥용)` 및 `자동 회랑(Corridor) 설정 완료`
  - 앱의 `🔄 서버 판결 도착: ACTION=KEEP` 및 `리스트 페이지 복귀 (유지)`

### 🚀 [STEP 7] "합짐" 2차 선점 (합짐 사냥 돌입 & 우회 동선 연산)
- **정의**: 첫콜을 문 상태(유지)에서 새로운 합짐 필터를 가지고 2차 콜을 낚아채는 고난이도 과정. 기존 동선과 새로운 동선을 병합(Waypoints)하여 우회 패널티를 계산합니다.
- **포함 대상 로그**:
  - 2번째 콜에 대한 1차 확정(`/confirm`) 통보 로그
  - 서버의 `상태: [합짐] 우회 동선 연산 돌입 (Waypoints Count: 3)`
  - 카카오 다중 경유지 계산 결과 `⚠️ 패널티 결과: +xx km, +xx분 '꿀'` 도출 로그

### 🚨 [STEP 8] 관제탑 무응답 및 데스밸리 방어기동 (예외 처리)
- **정의**: 관제탑이 일정 시간(30초) 결정을 내리지 못해 기사가 위험에 처하는 것을 방지하기 위해 스마트폰이 스스로 락다운을 풀고 취소 버튼을 누르는 탈출 기동입니다.
- **포함 대상 로그**:
  - 앱의 `⏳ 데스밸리 타이머 시작: 30초 대기...`
  - 관제웹과 서버의 `⚠️ [데스밸리 경고] 30초 경과 — 아직 판정 미결:`
  - 서버의 `💥 [서버 강제 해방] 35초 경과! 기기가 응답 대기 상태에 갇히는 것을 방지...`
  - 서버 측 `🚨🚨🚨 [EMERGENCY] 비상 보고 수신 (사유: AUTO_CANCEL)`

## 📝 자동 리포트 생성 스크립트 사용법 (`parser.js`)

매번 수작업으로 분할하고 페이로드를 매핑할 필요 없이, 정해진 폴더 구조 규칙을 유지하면 범용 **자동화 스크립트**([`parser.js`](file:///Users/seungwookkim/reps/onedal/log/parser.js))를 사용하여 완벽하게 정렬된 마크다운 리포트를 자동 생성할 수 있습니다.

### 1. 폴더 구조 준비
생명주기를 분석하고 싶은 시점의 이벤트들은 항상 `log/{월일_시간}` 형식 (예: `0416_1805`)의 이름으로 폴더를 생성하고, 그 내부의 `data` 폴더에 3개의 원본 로그 파일을 저장합니다.
```text
/Users/seungwookkim/reps/onedal/log/
 ├── parser.js               // 👈 범용 통합 파서 스크립트 (건드리지 말 것)
 └── 0416_1805/              // 👈 신규 생명주기 기록 폴더명 (이 이름으로 생성)
      └── data/
           ├── app.txt       // 안드로이드 앱 로그 (Android Logcat)
           ├── server.txt    // Node.js 서버 로그
           └── front.txt     // React 관제웹 콘솔로그
```

### 2. 스크립트 실행
터미널을 열고, 스크립트 옆에 대상 **폴더명**을 넘겨 실행합니다.
```bash
cd /Users/seungwookkim/reps/onedal/log
node parser.js 0416_1805
```

### 3. 결과 확인
명령어가 성공하면 대상 폴더 최상단에 **풀스택 분석 리포트**(`1DAL_Lifecycle_Report.md`)가 즉시 도출됩니다.
- 생성 위치 예시: `/Users/seungwookkim/reps/onedal/log/0416_1805/1DAL_Lifecycle_Report.md`
- **앱 시간대와 서버/웹 시간대(+9 KST) 자동 동기화 보정**이 이루어집니다.
- `[STEP 1]` ~ `[STEP 8]` 생명주기 분기가 로그 문맥을 인식하여 자동으로 이루어지며, 단절된 JSON Payload까지 누락 없이 결합합니다.
