# 기사님 제보 버그 원인 분석 및 해결 계획서

## 1. 현상 요약 (버그 증상)
* 기사님이 앱에서 직접 수동으로 콜을 터치하여 '확정' 페이지까지 진입함.
* 웹 관제탑에는 `[수동배차]`가 아니라 일반 매크로가 잡은 콜처럼 떴으며, 30초 데스밸리 타이머가 흐름.
* 기사님이 웹에서 `[유지(KEEP)]`를 눌렀음에도 불구하고, 앱과 연동되지 않고 30초 뒤 관제탑에서 콜이 증발함.
* 콜이 증발했는데 필터 상태는 이미 `[합짐 탐색중]`으로 강제 전환되어 있음.

## 2. 원인 분석 (로그 기반 100% 규명)

이 문제는 **"앱이 매크로(AUTO) 모드 켜져 있을 때, 사람이 손으로 콜을 누르면 발생하는 상태 불일치(Desync) 버그"** 입니다. 
로그를 시간대별로 추적한 결과, 원인은 다음과 같습니다.

### 🐛 문제의 발단: 앱의 거짓말 (19:14:01)
앱 로그: `📤 [post /confirm request] 서버 전송 내용 -> 모드: AUTO`
* 사람이 손으로 눌렀기 때문에 앱 내부의 **오토 사냥 엔진(`isAutoSessionActive`)은 꺼져 있는 상태**였습니다.
* 하지만 앱 좌측 하단의 스위치가 `[AUTO]`로 되어 있었기 때문에, 서버에는 **"제가 오토로 잡았습니다!" (`matchType: AUTO`)** 라고 거짓 보고를 보냈습니다.

### 💀 치명적 결함: 1.0초 통신 단절 (19:14:08 ~ 19:15:08)
* 서버는 오토가 잡은 줄 알고 "관제탑 결재(KEEP/CANCEL)를 대기하라"고 지시했습니다.
* 정상적인 오토 모드라면, 결재를 받기 위해 **1초에 1번씩 무전(Fast Polling)**을 쳐야 합니다.
* **그러나 앱 내부 오토 엔진(`isAutoSessionActive`)은 꺼져 있었기 때문에, "난 수동으로 잡은 거니까 결재 기다릴 필요 없지~" 하고 1.0초 고속 무전을 꺼버리고 60초 긴수면에 들어갔습니다.**
* `19:14:08` 이후 통신이 완전히 끊깁니다.

### ⏳ 관제탑의 결재와 엇갈림 (19:14:27)
프론트엔드 로그: `[🖥️관제웹] PinnedRoute에서 KEEP(유지 확정) 버튼 클릭`
* 기사님이 웹에서 19분 14초 27초경에 수동으로 `[KEEP]` 버튼을 눌렀습니다.
* 서버는 "오! 기사님이 결재했다! 빨리 앱폰에 전달해!" 라며 Piggyback 큐에 넣었습니다.
* 동시에 콜이 확정되었으므로 **상태를 `EMPTY`에서 `LOADING(합짐 탐색중)`으로 전환**시켜 버립니다. (기사님이 보신 합짐 탐색중의 원인)
* 하지만 앱은 60초 수면 중이라 서버의 이 무전을 받지 못합니다.

### 💥 비극의 끝: 데스밸리 타임아웃 (19:14:44)
서버 로그: `🔴 [웹 수신] order-canceled`
* 서버는 결재를 보냈는데 앱이 30초가 넘도록 아무 대답이 없자(수면 중이므로), 통신 두절/에러로 간주하고 **보호 조치를 위해 콜을 취소 처리하고 관제탑에서 삭제**해버렸습니다.

## 3. 해결 계획 (Proposed Changes)

이 문제를 근본적으로 해결하기 위해 **안드로이드 앱의 텔레메트리 판별 로직**을 수정해야 합니다.

### [MODIFY] 안드로이드 앱 `HijackService.kt`
* **수동/자동 완벽 분리**: 콜을 눌렀을 때, 우측 하단 스위치(`currentMode`)가 `AUTO`이건 `MANUAL`이건 상관없이, **진짜 매크로가 클릭한 게 아니라면(`isAutoSessionActive == false`) 서버에 무조건 `matchType = "MANUAL"`로 솔직하게 보고**하도록 수정합니다.
* 이렇게 되면 사람이 누른 콜은 서버가 묻지도 따지지도 않고 즉결 수동 확정 처리(`Action: KEEP`)를 내리게 되며, 30초 데스밸리도 발생하지 않고 1.0초 무전이 꺼져 있어도 완벽하게 동작합니다.
