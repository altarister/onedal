# 필터 설정 UI 통합 및 파이프라인 연동 계획서

현재 기사님께서 지적하신 바와 같이, `OrderFilterModal.tsx`가 `isSharedMode`(합짐 여부)에 따라 입력창을 동적으로 숨기고 나타나게 만들어져 있어 **"사전에 모든 시나리오(첫짐 + 합짐)에 대한 설정을 미리 세팅해 둘 수 없는 치명적인 단점"**이 존재합니다. 

이를 해결하고 공통 로직으로 직관적으로 동작하도록 아키텍처와 UI를 대폭 수정합니다.

## ⚠️ User Review Required

본격적인 개발에 앞서 아래 기획안이 사장님의 의도와 정확히 일치하는지 확인 부탁드립니다.

> [!IMPORTANT]
> **핵심 변경 사항 요약**
> 1. 모달창 내에서 '단독/합짐' 모드 전환 시 UI가 사라지는 현상을 없애고, **3개의 구역(공통/첫짐/합짐)으로 상시 노출**시킵니다. (✅ 완료)
> 2. 하드코딩되어 있던 회랑 반경(10km) 로직을 제거하고, 기사님이 설정창에서 미리 세팅해 둔 `corridorRadiusKm` 값을 합짐 전환 시 자동으로 끌어다 쓰도록 서버 로직을 변경합니다.

## 📱 안드로이드 앱 전송 필터(AutoDispatchFilter) 스펙 정리

서버의 설정 데이터에는 UI 렌더링용 데이터(목표 도시명, 아코디언 그룹핑 등)가 섞여 있지만, **실제 안드로이드 앱이 0.01초 필터링을 위해 사용하는 핵심 키값**만 추려보면 첫짐/합짐 구분 없이 아래의 **단일 JSON 구조**로 완벽하게 귀결됩니다.

앱은 자기가 지금 "첫짐"을 잡는지 "합짐"을 잡는지 모릅니다. 서버가 아래 JSON의 알맹이만 계속 갈아끼워주면, 앱은 묵묵히 아래 5가지 조건만 무지성으로 검사합니다.

```json
{
  "isActive": true,
  "loadState": "EMPTY",
  "isSharedMode": false,

  // [공통 검사 로직]
  "allowedVehicleTypes": ["1t", "다마스", "라보", "오토바이"],
  "minFare": 35000,
  "maxFare": 1000000,
  "excludedKeywords": ["착불", "까대기", "왕복", "급송"],
  "pickupRadiusKm": 15,          // 기사 위치 - 오더 상차지 간의 직선 거리 (앱이 직접 계산)

  // [도착지 검사 로직 - 정적 지역 기반]
  "destinationCity": "파주시", 
  "destinationRadiusKm": 5,      
  "destinationKeywords": [
    "금촌동", "교하동", "운정동", "문산읍", "파주읍", "법원읍", ... // 파주시에 속한 50개 읍면동
  ]
}
```
```json
{
  // 1. 매크로 가동 여부
  "isActive": true,

  // 2. 차종 검사 (오더의 차량 종류가 배열에 있는가?)
  "allowedVehicleTypes": ["1t", "다마스", "라보", "오토바이"], 

  // 3. 단가 검사 (오더 단가가 최소/최대 범위 안에 있는가?)
  "minFare": 35000,
  "maxFare": 1000000,

  // 4. 지뢰콜/블랙리스트 검사 (오더 텍스트에 아래 단어가 없는가?)
  "excludedKeywords": ["착불", "까대기", "왕복", "급송"],

  // 5. 상차지 거리 검사 (기사 위치에서 오더 상차지까지의 직선거리가 이내인가?)
  "pickupRadiusKm": 15,          

  // 6. 목적지 방향성 검사 (오더의 하차지 동이름이 아래 50개 텍스트 중 하나인가?)
  "destinationKeywords": [
    "서초동", "양재동", "판교동", "금곡동", "정자동", "수지구", "분당구", ...
  ]
}
```

> [!NOTE]
> **서버의 역할 (Brain)**: "첫짐"일 때는 목표 도시의 읍면동 50개를 `destinationKeywords`에 넣어주고, "합짐"일 때는 기사가 가는 길(회랑)의 읍면동 50개를 계산해 `destinationKeywords`에 넣어줍니다.
> **앱의 역할 (Muscle)**: 서버가 준 `destinationKeywords` 안에 오더의 목적지 텍스트가 포함되어 있는지만 검사합니다.

# 필터 시스템 아키텍처 전면 재설계안 (피드백 반영)

사장님께서 남겨주신 코멘트들을 쭉 읽어보니, 현장의 도메인 지식(Domain Knowledge)이 완벽하게 녹아있는 **가장 이상적인 방향**입니다. 지적하신 내용들이 백번 맞습니다.

기존의 "공통 설정"이라는 애매한 개념을 버리고, 사장님의 의견을 100% 반영하여 시스템을 두 가지 차원(DB 기반 디폴트 vs 메모리 기반 일회성)과 두 가지 상태(첫짐 vs 합짐)로 완전히 분리 재설계합니다.

---

## 1. 🎯 안드로이드 앱 전송 스펙 확정 (피드백 정리)

앱은 정말 "수동적인 필터 머신"으로만 동작하며, 서버가 상황에 맞게 값을 치환해서 내려줍니다.

| 필터 항목 | 첫짐(EMPTY) 시 서버가 앱에 주는 값 | 합짐(LOADING/DRIVING) 시 서버가 앱에 주는 값 | 사장님 코멘트 반영 결과 |
| :--- | :--- | :--- | :--- |
| **차종** (`allowedVehicleTypes`) | 기사가 설정한 원본 차종 (예: `["1t", "다마스", "라보"]`) | **서버가 큰 차종 자동 제거** (예: `["다마스", "라보"]`만 남김) | "합짐이 생길수록 작은 단위로 바뀌어야 함" ✅ |
| **하한가** (`minFare`) | **첫짐용 하한가** (예: 35,000원) | **합짐용 하한가** (예: 15,000원 - 거리가 짧아지므로) | "합짐은 거리가 짧아 단가가 작아져야 함" ✅ |
| **상차 반경** (`pickupRadiusKm`) | 기사가 설정한 상차 반경 (예: 15km) | **무시 (`999km` 등 강제 할당)** | "합짐일 때는 어차피 가는 길이니 상차반경 불필요" ✅ |
| **하차지 읍면동** (`destinationKeywords`) | **목표 도시(파주시)의 50개 읍면동** | **가는 길(회랑) 근처의 50개 읍면동** | "상황에 따라 서버가 계산해서 바꿔치기" ✅ |
| **블랙리스트** (`excludedKeywords`) | 기사가 설정한 블랙리스트 | 동일 | 유일한 진짜 공통 항목 |

*(참고: `maxFare`, `isActive`, `destinationCity`, `destinationRadiusKm` 등은 앱의 필터링 엔진 JSON에서 아예 빼버리겠습니다. 서버에서만 씁니다!)*

---

## 2. 🗄️ 설정의 이원화 (디폴트 DB vs 세션 메모리)

말씀하신 대로 "매일 출근 시 불러오는 기본값(DB)"과 "오늘 이 순간만 쓸 일회성 세팅(Memory)"이 섞여 있어서 혼란이 왔습니다. 이를 UI 상에서 명확히 분리합니다.

### A. 기사 프로필 기본 설정 (DB 영구 저장)
*   **위치**: 우측 상단 톱니바퀴 `[환경 설정 모달]` 내 **'배차 기본 세팅'** 탭 신설
*   **용도**: 매일 출근해서 앱을 켤 때 기본으로 깔리는 값들.
*   **항목**:
    *   내 기본 차량 타입 (나는 보통 1t과 다마스를 몬다)
    *   나만의 블랙리스트 키워드 (착불, 수거 등 영구 제외)
    *   첫짐 최소 단가 마지노선 (예: 무조건 3.5만 이상)
    *   합짐(잔짐) 최소 단가 마지노선 (예: 합짐은 1.5만 이상)

### B. 현재 운행 세션 설정 (서버 메모리 - 휘발성)
*   **위치**: 현재 관제 대시보드 화면 중앙의 `[필터 설정]` (OrderFilterModal)
*   **용도**: "지금 당장 어디로 갈 것인가?"를 정하는 이번 주행 전용 1회성 세팅.
*   **항목**:
    *   **첫짐 탭**: 도착 희망 시/도 (예: 오늘은 파주로 가자!), 상차 반경 (오늘은 10km 이내만 잡자)
    *   **합짐 탭**: 적재 중 경로 이탈 허용 반경 (오늘은 합짐 5km까지만 우회하자)
*   **작동 방식**: 여기서 값을 바꾸면, **A(기본 설정)**을 베이스로 깔고 **B(현재 세팅)**가 덮어씌워진 상태로 서버 메모리에 저장되어 배차가 돌아갑니다. 퇴근 후 서버가 꺼지거나 새로고침하면 이 값은 초기화되고 다시 A(기본 설정)를 불러옵니다.
