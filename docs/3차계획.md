# 1DAL 노쑈(No-Show) 방지 안전 모드 설계안 v3 (최종 진화형)

> **핵심 원칙**: 안드로이드 앱은 그저 "눈과 손(Dumb Client)"이며, 무슨 판단을 해야 하는지는 오직 "뇌(Server)"가 통제한다.

대표님의 피드백을 수용하여 아키텍처를 전면적으로 개편합니다.
앱에 존재하던 하드코딩된 로직(12초 등)과 인성앱 디펜던시를 모두 덜어내고, 서버 중심의 제어권(Smart Server)으로 이관합니다.

---

## 🏗️ 아키텍처 패러다임 전환 (Smart Server & Dumb Client)

### 기존의 한계
* **앱이 너무 똑똑함**: 앱 스스로 "내가 12초 기다렸으니 초기화해야지", "지금 취소 버튼 누를게" 라고 독단적으로 판단.
* **디펜던시**: 앱 안에 "여기 인성앱 확정화면 맞지?" 식의 로직이 하드코딩되어 확장이 어려움.
* **관제 불가**: 폰이 지금 무슨 화면을 보고 있는지 서버와 대표님은 알 길이 없음.

### 💡 V3 개편 방향 (대표님 아이디어)
1. **앱의 역할 (Scraper & Clicker)**: 화면이 바뀔 때 마다 `[화면 안의 텍스트 전부]` + `[앱이 추정하는 현재 화면 이름]`만 묶어서 서버로 쏩니다.
2. **서버의 역할 (Brain)**: 앱이 쏜 텍스트를 분석하여, "지금 오더가 꼬였네? 긴급 취소해!", "계속 대기해" 명령을 응답으로 내려줍니다.
3. **관제탑 (Dashboard)**: 서버가 파악한 각 앱폰의 "현재 시청 중인 화면"을 `telemetry-panel`에 실시간 텍스트로 중계합니다.

---

## 📱 Layer 1. 앱폰의 진화 (화면 상태 실시간 리포트)

### 1) 12초 하드코딩 제거 및 타이머 자율성
* `HijackService.kt:98`에 있던 12초 하드코딩을 삭제합니다.
* 앱은 "시간 타이머" 자체를 관리하되, 혼자서 상태를 초기화(Searching)하지 않습니다. 
* **"서버 응답 제한시간(30초)"** 안에 "닫기(KEEP)" 응답을 받지 못하면 무조건 취소 버튼을 찾아서 클릭합니다. (안전우선)

### 2) 화면 상태(Screen Status) 리포터 추가
* **동작 방식**: 텔레메트리나 API 요청 시, 항상 기기가 현재 보고 있는 화면 상태를 `screenContext` 라는 필드에 담아 보냅니다.
* **ScreenContext Enum (예시)**:
  - `LIST` : 사냥 리스트 화면
  - `DETAIL_PRE_CONFIRM` : 콜 삼키기 전(광클 찰라)
  - `DETAIL_CONFIRMED` : 확정 완료된 상세 화면
  - `POPUP_PICKUP` : 상차지/하차지 팝업
  - `UNKNOWN` : 그 외 알 수 없는 화면 (다른 앱이 켜졌거나, 길도우미 등)

### 3) 행동 강령 (Dumb Client)
* 서버가 `POST /detail`의 롱폴링 응답으로 `{"action": "KEEP"}`을 보내면 닫기를 누름.
* 응답이 없어서(통신단절) 스스로 30초가 다 되면 취소 누름.
* **"취소 버튼 못 찾음" 이나 "이상한 화면 진입" 발생 시**:
  - 즉각적으로 서버에 `POST /emergency` 와 함께 현재 텍스트 전부 전송.
  - 서버 판단 대기.

## 🧠 Layer 2. 서버의 진화 (상황실 & 인공지능)

### 1) 다중 앱 (인성/24시 등) 설정 외부 분리
* `server/config/targets.json` 을 신설.
* 각 타겟 앱(inSung, 24h)별로 어떤 텍스트가 나오면 LIST이고 어떤 텍스트가 CONFIRMED인지 정의.
* **효과**: 인성앱 업데이트로 UI 글자가 바뀌면, 앱 재설치 없이 서버 `targets.json` 파일만 고치면 즉각 해결.

### 2) V3 타임아웃 & 비상 복구 처리
* 앱은 롱폴링(`POST /detail`) 대기 중입니다.
* 서버에서 판단(카카오 연산)하고 대표님 결정이 지연되어 30초에 임박한다면 (ex: 28초):
  - 서버는 그냥 가만히 있습니다.
  - 앱폰이 30초가 되는 순간 "어? 서버 대답 없네(닫으란 적 없네). 무조건 취소 쳐!" 라며 인성앱 `[취소]` 클릭.
* 앱이 취소를 누르려는데 "시간이 지나 취소 불가" 팝업이 뜬다면:
  - 앱 : 텍스트 스크랩 ➡️ 서버로 `[이상상황 송신]`
  - 서버 : 아, 이거 취소불가 팝업이구나 ➡️ 관제탑에 **"🚨 수동 전화 취소 요망"** 경고 알림. (이때만 대표님이 개입)

### 3) 2폰 동시 낚임 경합 로직 강화
* `detail.ts` 의 본콜/합짐 메모리.
* 아직 관제탑(대표님)이 허락 안 했는데, 앱폰2가 합짐 삼켜서 가져오면?
* 서버: "잠깐 대기. 관제사 승인 할 때까지 너도 데스밸리(30초) 대기해".

---

## 🖥 Layer 3. 관제탑 (Dashboard)

### 1) Telemetry Panel 고도화 (대표님 요청)
* 현재 패널에는 연결 상태(수집/수락/취소)만 뜹니다.
* 여기에 기기가 현재 보는 화면: `[인성앱-리스트]`, `[인성앱-확정화면]`, `[알수없음]` 을 배지로 표시합니다.
* 기성품처럼 대표님이 앱 화면을 안 보고도 기기가 뻘짓하고 있는지 즉각 모니터링 가능.

### 2) 수동 조치 가시화
* 취소 버튼 소실, 에러 팝업 등 발생 시 관제탑 오더 카드에 삐용삐용 붉은 알람 노출.
* "앱이 취소 불가 상태입니다. 배차실에 전화해서 치워주세요!" 메시지 노출.

---

## 📋 요약: 1분이 지날 수 없는 완벽한 데스밸리 방어벽

1. **(제한선)** 배차 후 28~30초엔 무조건 취소 버튼이 눌립니다. (앱 내부 타이머)
2. **(서버 통제)** 서버가 명시적으로 "확정(닫기)" 명령을 주기 전까지는 데스밸리 가동 구간으로 봅니다.
3. **(관제 지원)** 만약 꼬여도(이상한 팝업, 취소 버튼 실종) 폰은 즉시 화면을 찍어서 서버로 쏘고, 서버는 대표님 모니터로 위험 노티를 때립니다.

이렇게 서버로 비즈니스 로직을 몰아넣는 아키텍처에 동의하시면 바로 코딩 짜겠습니다!
