# 1DAL Platform: 스마트 회랑(동적 합짐) 최종 기획 및 아키텍처

대표님과 논의하며 나온 인사이트들을 집대성한 최종 기획입니다. 모든 생각(첫짐/합짐 파이프라인, 고속도로 톨게이트 딜레마, UI 압축)이 완벽히 동기화되었습니다. 아래는 향후 개발을 위한 완벽한 청사진(Blueprint)입니다.

---

## 1. 코어 파이프라인: 첫짐 ➡️ 합짐 필터 전환 (Workflow Sync)
우리(서버)와 기사(앱)가 동작하는 흐름은 다음과 같이 완벽히 일치합니다.
1. **[빈차 상태] - 첫짐 필터 (수동)**: 기사는 원하는 도착지, 금액을 앱에 수동으로 세팅하고 대기합니다.
2. **[기둥콜 픽업]**: 기사가 조건에 맞는 첫 콜(본콜)을 잡고 '확정' 버튼을 누릅니다.
3. **[서버 회랑 계산]**: 서버는 해당 콜의 출/도착지를 기반으로 카카오 다이렉션 API를 찔러 **주행 경로(Polyline)**를 얻고, Turf.js를 이용해 이 선과 겹치는 **'스마트 회랑 지역 목록(합짐 필터)'**을 생성합니다.
4. **[S2S 필터 주입]**: 앱이 주기적인 `/api/scrap` 통신을 보낼 때, 서버는 응답값의 `dispatchEngineArgs` (도착지 목록)에 방금 계산한 '합짐 필터'를 내려보냅니다.
5. **[앱 자동 덮어쓰기]**: 앱은 이 값을 받는 즉시 기존 '첫짐 필터'를 날려버리고, 서버가 준 **'합짐 필터 (스마트 회랑)'로 레이더를 100% 바꿉니다.**
6. **[합짐 스캐닝]**: 이제 앱은 철저하게 내가 주행할 경로 상의 읍면동에서 떨어지는 콜만 낚아채어 서버로 올립니다.

## 2. 심화 고민 및 해결책 (Deep Dive)

### 2.1 고속도로 톨게이트 딜레마 (Tolled Highway Detour Problem)
> **대표님 코멘트**: *"고속도로는 톨게이트를 통해서 나갔다 들어왔다 해야 하니까 인접해 있다 하더라도 합짐하면 너무 많이 돌 수 있을것 같다. 어딘가에 고민을 저장해두자."*

**저장 및 향후 로직 방향성:**
정말 예리한 지적입니다. 직선 거리만 보면 반경 5km 안에 콜이 있어도, 그곳이 고속도로 구간이면 톨게이트로 나갔다 들어오는 데만 30~40분이 버려집니다.
*   **향후 고도화 방안 식별**: 
    1. 카카오 API의 `guides` 정보에서 해당 구간의 `road_type`이 '고속국도'나 '도시고속도로'인 구간은 **반경(Buffer) 연산에서 제외(Radius = 0)시키거나 대폭 축소**합니다. 
    2. 오직 국도/지방도 전환점이나 'OO 톨게이트(IC)' 근방 반경에서만 합짐을 스캔하도록 Turf.js 공간 연산 레벨을 고도화할 수 있습니다. (현재는 기록으로 남기고, 추후 버전에서 적용)

### 2.2 차량 이동 상태 자동 인식 (GPS 기반 Auto-Strict)
> **대표님 코멘트**: *"운전 중인지 아닌지는 핸드폰 GPS를 이용하는 거지?"*

**작동 원리**: 
*   맞습니다. 앱은 `/api/scrap`이나 백그라운드에서 주기적으로 기기의 GPS 위경도, 혹은 속도 변화를 서버에 전송합니다(또는 전송하도록 보완합니다). 
*   서버는 이 좌표의 변위를 계산하거나 속도를 확인하여, 차량이 명확히 '주행 상태(이동 시작)'로 판단되면, 폰의 세팅이 광역(5/10km)이었더라도 **강제로 반경 0km (인접-정밀 모드)** 파라미터로 즉시 덮어씌워 폰에 다시 쏴줍니다. 

### 2.3 경로 옵션 (다중 루트) 선택 
*   카카오 API(`priority=RECOMMEND, TIME, DISTANCE`)로 받아온 대안 경로는 **'첫짐'을 잡을 때 (루트를 선정할 때) 1회만 사용**됩니다.
*   기사가 "나는 서울 관통 안 하고 외곽으로 돌거야" 하고 루트를 픽스하면, 그 궤적이 **합짐 회랑의 뼈대**가 되며, 이후 운전 중엔 이 뼈대가 변하지 않습니다.

## 3. UI/UX 디자인 가이드

현업의 시인성과 퍼포먼스를 해치지 않기 위해 텍스트 나열과 복잡한 지도를 피합니다.

### 3.1 TBT 텍스트 축약 표시 (Turn-by-Turn)
> **대표님 코멘트**: "도로명을 적당히 축약해서 (남양로 - 오름지하차도 - 옥구천동로) 이정도만 한줄로 표시해 주어도 좋겠다."

카카오 API의 `guides`를 파싱할 때 '좌회전', '우회전' 같은 쓸데없는 텍스트는 정규식으로 버리고, **주요 도로명이나 랜드마크(IC/JC)만 추출**하여 우아하게 한 줄로 압축합니다.

### 3.2 대시보드 상태창 UI (Seamless Integration)
기존 필터 UI 포맷(`예: 일반 | 4 | 30km | 광주시(10km)`)을 망가뜨리지 않고 자연스럽게 스며들도록 구성합니다.

**[ UI 예시 - 리스트 뷰 ]**
```text
(기존) 단독 | 4 | 30km | 성남시, 광주시 (수동 타겟)
(변경) 회랑 | 정밀(0km) | 중부고속 - 경충대로 - 옥구천동로
(변경) 회랑 | 광역(5km) | 영동고속 - 남양로 - 송산지하차도
```
*   **리스트 뷰 vs 캔버스(지도) 뷰**: 
    리스트 위주의 빠른 텍스트 확인(퍼포먼스 최고)을 기본 화면으로 가져가고, 만약 기사나 관제사가 실제 지형을 눈으로 꼭 확인하고 싶을 때만 **선택적 모달(지도 아이콘 클릭 시 Canvas Open)**로 궤적을 렌더링하는 투트랙 UX가 성능에 유리합니다.
