## 🔴 긴급 버그: CANCEL 수신 후 앱이 확정페이지에 멈추는 문제

### 원인 분석

서버에서 `{"action":"CANCEL"}`을 내려줘서 앱이 `EXECUTING_DECISION` 상태로 전환되긴 합니다. 하지만 **이 버그의 핵심 원인은 `onAccessibilityEvent`가 `TYPE_WINDOW_CONTENT_CHANGED` 이벤트에서만 실행된다는 것**입니다:

```
1. 서버 CANCEL 응답 수신 (백그라운드 스레드)
2. mainHandler.post { serverDecision = "CANCEL"; transitionTo(EXECUTING_DECISION) }
3. → 이제 "취소" 버튼을 눌러야 하는데...
4. → onAccessibilityEvent()는 화면 내용이 바뀌어야만 호출됨
5. → 확정 페이지가 그대로 정지해있으면 이벤트 자체가 안 옴!
6. → "취소" 버튼 터치가 영원히 실행되지 않음 ❌
```

즉, `transitionTo(EXECUTING_DECISION)`만으로는 즉시 행동이 발생하지 않습니다. 다음 화면 변경 이벤트가 올 때까지 아무 일도 안 합니다.

### 수정 방안

**`EXECUTING_DECISION` 전환 시, 즉석에서 `rootInActiveWindow`를 잡아 바로 취소/닫기 버튼을 누르는 즉시 실행 함수를 추가합니다.**

#### [MODIFY] [HijackService.kt](file:///Users/seungwookkim/reps/onedal/onedal-app/app/src/main/java/com/onedal/app/HijackService.kt)

1. 새 함수 `executeDecisionImmediately()` 추가:
   - `rootInActiveWindow`에서 직접 "취소" 또는 "닫기" 를 찾아 터치
   - 성공하면 `SEARCHING`으로 전환 + `currentOrder = null`
   - 실패하면 그냥 리턴 (다음 `onAccessibilityEvent`에서 재시도)

2. 기존 `sendDetail` 콜백 (line 392-398)에서 `transitionTo(EXECUTING_DECISION)` 바로 뒤에 `executeDecisionImmediately()` 호출

3. 데스밸리 자동취소(line 154-162)에서도 동일하게 `executeDecisionImmediately()` 호출

> [!IMPORTANT]
> 이 수정이 안 되면, CANCEL을 받아도 앱이 **영원히** 확정 페이지에 멈출 수 있습니다. 다음 스크래핑 타임아웃(15초)이 발동해서야 겨우 `SEARCHING`으로 복귀하는 우회 경로밖에 없습니다.

## 카카오 추천 로직(추천 UI) 관련 설명

> [!NOTE]
> **추천 로직은 이미 구현되어 있습니다!** 대시보드의 PinnedRoute 컴포넌트에 "카카오 동선 최적화 결과" 패널이 뜨는 것이 바로 그것입니다 (스크린샷에 보이는 "+60.1km 추가 주행", "기존 대비 +59분 추가 소요").
> 
> 다만, 이 정보는 **서버가 카카오 API를 호출하여 계산한 결과**이며, 별도의 "잡을지 말지 자동 추천" AI 로직은 아닙니다. 현재는 관제사(대표님)가 이 데이터를 보고 **수동으로 방출/유지를 판단**하는 구조입니다.
> 
> 향후 "수익률이 X% 이상이면 자동 수락, 우회 시간이 Y분 초과면 자동 방출" 같은 자동 판단 엔진을 추가하려면 별도 설계가 필요합니다. 이번 작업 범위에 포함시킬까요?
