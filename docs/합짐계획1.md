# [스마트 회랑 추출] 동적 합짐(On-The-Way) 필터링 구현 기획서

PRD 6번에 명시된 **초정밀 노선 기반 합짐 상차 필터링**을 구현하기 위한 설계 및 기획서입니다. 목표는 본콜이 확정되었을 때 기사가 이동할 **"경로(도로)"를 파악**하고, 그 경로에 해당하는 읍/면/동을 자동으로 추출해 **동적으로 필터링**하는 것입니다.

대표님 말씀대로 읍면동 자체를 리스트업해서 화면에 보여주지 않고, 서버 단에서 투명하고 자동적으로 처리하는 구조로 설계했습니다.

---

## 1. 개요 및 구현 전략 (How it works S2S)

"도로명 ↔ 읍면동" 데이터베이스를 우리가 엑셀 수작업으로 구축하는 것은 막대한 공수가 듭니다(전국 도로망). 따라서 **카카오 리버스 지오코딩 API의 응집력**을 활용한 **'경로 위 좌표 샘플링 기법'**을 제안합니다.

### 🔄 핵심 워크플로우

1. **본콜 확정 (Trigger)**: 관제탑 대시보드나 앱에서 첫 콜(본콜)을 "확정" 상태로 만듭니다.
2. **카카오 내비 경로 추출 (Route Coordinates)**: 백엔드 노드가 카카오 모빌리티 API를 호출하여, 상차지 ➡️ 하차지까지의 "주행 궤적(수천 개의 X,Y 좌표 배열)"을 받아옵니다.
3. **스마트 간격 샘플링 (Sampling)**: 50km 주행일 경우, 주행 궤적을 3~5km 간격으로 등분하여 약 10~15개의 "주요 경로 위 좌표"를 찍어냅니다. (휴게소, IC, 주요 도로 위).
4. **리버스 지오코딩 (Reverse Geocoding)**: 추출된 15개의 좌표를 카카오 `coord2regioncode`(좌표 ➡️ 행정동 변환) API에 S2S로 동시(병렬)로 던집니다.
5. **동적 필터 생성 체인 (Dynamic Injection)**: 리턴받은 행정동(삼평동, 태평동 등)의 중복을 제거한 뒤, **`activeFilterConfig`의 `targetRegions`에 남몰래 강제 주입**합니다.
6. **지뢰 제거 (Filtering)**: 1초 뒤 앱폰이 바뀐 필터를 동기화 받고, 이제 해당 '지리 회랑' 안에 드는 오더만 칼같이 클릭합니다.

## 2. 세부 컴포넌트 설계

### 2.1 Backend (Node.js)

#### [NEW] `kakaoUtil.ts - extractCorridorRegions()`
본콜의 출발/도착 좌표를 받아 다음 연산을 수행합니다:
- `directions.json` API로 경로(`vertexes` 좌표 배열) 획득
- 전체 배열에서 20개 정도 평등한 간격으로 Index를 추출
- 병렬 `Promise.all`로 20개의 좌표를 Kakao 로컬 API에 던져 `region_3depth_name` 파싱
- Array Array에서 중복 배제 (e.g., `["반정동", "영통동", "원천동", "상현동"]`)

#### [MODIFY] `detail.ts` & `emergency.ts`
- 관제사가 오더를 **KEEP (본콜 확정)** 하는 순간 `extractCorridorRegions()`를 백그라운드로 돌립니다.
- 결과가 나오면 `filterStore`의 모드를 `합짐`으로 자동 전환시키고, `targetRegions`를 이 배열로 덮어씌운 뒤 클라이언트에 HMR(Socket)로 뿌립니다.

## 3. 화면 처리 기준 (UI/UX)

대표님의 요청: **"읍/면/동은 화면에 표현하지 않을 꺼야."**
- 기존 대시보드 좌측 패널(필터 상태창)에 `[타겟 지역: XX동, YY동...]` 이라고 나열하지 않습니다.
- 대신, 본콜이 확정되면 대시보드 필터 상태창의 디자인을 **"경로 기반 자동 모드(Route Mode)"** 전용 패널 형식으로 전환합니다.

**[ UI 예시 - 대시보드 필터 컨트롤러 ]**
```text
🎯 합짐 스니핑 활성화됨
-----------------------
[ 🛣️ 경부고속도로 경로 추적 중 ]  <-- (내부적으로 대표 고속도로명/국도명 1개 정도만 대표 텍스트로 노출)
[ 🧭 본콜 이동 궤적 내 18개 읍/면/동 자동 필터링 중... ]  <-- (건수만 표기)
-----------------------
```
이러면 화면이 난잡해지지 않으면서, 관제사가 "시스템이 길목을 잘 지키고 있구나"를 한눈에 확신할 수 있습니다.

## 4. User Review Required (피드백 요청 사항)

개발에 돌입하기 전 아래 정책에 대해 의사결정이 필요합니다.

> [!IMPORTANT]
> **확장 반경 문제 (Adjacency)**
> 경로 상 좌표에 정확히 겹친 동(예: 반포동)만 가져오게 됩니다. 바로 옆에 붙어있는 "잠원동"은 도로가 관통하지 않았다면 리스트에서 빠집니다.
> **옵션 A**: 일단 도로가 통과하는 "정확한 읍/면/동" 만 칼같이 가져와서 초정밀 회랑만 구성한다. (너무 좁을 수 있음)
> **옵션 B**: 해당 시군구 라벨(예: 서초구 전체)까지 보조로 느슨하게 가져온다. (효율 하락)
> **(추천 옵션 A)**: 합짐의 첫 목적인 "시간지연 없는 극강의 동선"을 달성하려면 보수적인 옵션 A가 가장 PRD에 부합합니다.

> [!WARNING]
> **필터 덮어쓰기 정책**
> 본콜 확정 시 기존에 설정되어 있던 수동 지역 필터(targetCity 등)는 **무시되고** 곧바로 스마트 회랑 필터로 **완전히 덮어씌워져도** 될까요? (이렇게 하는 것이 자연스럽습니다.)

의사결정 피드백(혹은 "옵션 A로 오케이, 진행해") 주시면, 곧바로 카카오 지오코딩 모듈과 자동화 로직 코딩을 시작하겠습니다!
