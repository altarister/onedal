# [스마트 회랑 추출] 동적 합짐(On-The-Way) 필터링 구현 기획서

PRD 6번에 명시된 **초정밀 노선 기반 합짐 상차 필터링**을 구현하기 위한 설계 및 기획서입니다. 목표는 본콜이 확정되었을 때 기사가 이동할 **"경로(도로)"를 파악**하고, 그 경로에 해당하는 읍/면/동을 자동으로 추출해 **동적으로 필터링**하는 것입니다.

---

## 1. 용어 정의 및 타입 표준화 (Terminology)
기획 단계부터 개발 코드(TypeScript) 레벨까지 용어를 완벽하게 통일합니다.
*   **첫짐 (FirstLoad)**: 차량이 완전히 비어있는 상태(예: 아침 출근)에서 처음으로 잡게 된 기둥콜.
*   **합짐 (CombinedLoad / Piggyback)**: '첫짐'의 이동 경로(Corridor)를 훼손하지 않으면서 이동 중 추가로 싣게 되는 콜.
*   **타입스크립트 정의**:
    ```typescript
    type DispatchMode = '첫짐' | '합짐';
    ```

---

## 2. 덮어쓰기 정책 및 필터 모드 자동화
*   **완전 덮어쓰기 (Override)**: 사용자가 "스마트 합짐 모드"를 켜는 순간, 기존에 폰에 남아있던 수동 타겟 지역 설정은 증발하고 **서버가 제공한 스마트 회랑 지역 목록으로 완전히 덮어씌워집니다.**
*   **주행 상태 자동 인식 (Auto-Strict)**: GPS 변위를 통해 차량의 등속 실주행(이동)이 감지되는 순간, 시스템은 오직 **[인접(0km) 정밀 모드]**로 필터를 자동 축소하여 주행 중 노선 이탈을 원천 차단합니다. 

---

## 3. 화면 처리 기준 (UI/UX)
대표님의 요청: **"단 하나의 줌 없는 캔버스로 심플하게 표현하고, 상세 읍/면/동은 노출시키지 마라"**

### 3.1 PinnedRoute (합짐 확정 패널 UI)
*   사용될 컴포넌트: `components/dashboard/PinnedRoute.tsx`
*   **단일 고정 캔버스 (Static Map Canvas)**:
    *   줌(Zoom) 기능이나 패닝 기능이 없는 정적 캔버스로 렌더링.
    *   상차지(1) ➡️ 하차지(2) 등을 상징하는 동그란 핀(Pin) 마커만 경로상에 표기.
*   **경로 선택 옵션**: 카카오 방향 API(`priority`)의 대안 경로가 있을 때, 캔버스 아래 대안 버튼을 눌러(예: `[경유: 서울시내]`, `[우회: 외곽순환]`) 노선을 스왑해 볼 수 있습니다. (※ 이 노선 탐색 기능은 '첫짐'을 잡을 때만 제공하고 확정하면 고정됩니다.)

### 3.2 OrderFilterStatus (대시보드 필터 컨트롤러 UI)
*   사용될 컴포넌트: `components/dashboard/OrderFilterStatus.tsx`
*   기존에는 `[첫짐 | 4만 | 30km | 성남시(10km)]` 형태로 보였습니다.
*   "합짐" 모드로 넘어오면 카카오 `guides` 정보에서 추출한 가장 큰 도로명 위주로 텍스트를 축약 교체합니다.

**[ 상태창 UI 텍스트 변경 포맷 ]**
```text
(첫짐 모드) 🟩 첫짐 | 4.0 | 30km | 성남시(10km)     << 수동 세팅
(합짐 모드) 🟪 합짐 | 광역(5km) | 영동고속 - 남양로    << 서버 세팅
(합짐 운행) 🟪 합짐 | 정밀(0km) | 남양로 - 송산지하차도 << 자동 전환
```

## 4. 첫짐 / 합짐 JSON 페이로드 구조 (Payload Specs)
서버가 앱에 던져주는 필터 제어 명령의 구조입니다. `/api/scrap`의 리스폰스인 `dispatchEngineArgs` 규격입니다.

### 4.1 첫짐 (첫짐스캔 대기중 일 때) - 앱 수동 의존
```json
{
  "mode": "첫짐",
  "minFare": 40000,
  "pickupRadius": 30,
  "targetCity": "성남시",
  "targetRadius": 10,
  "targetRegions": [],  // 앱이나 기사가 세팅한 대로 동작하도록 빈배열이나 null
  "blacklist": []
}
```

### 4.2 합짐 (합짐 노선 스캔 중일 때) - 서버 강제 제어
앱은 이 JSON을 받는 순간 자신이 알던 룰을 파괴하고, `targetRegions`에 포함된 콜만 무조건 잡아올립니다.
```json
{
  "mode": "합짐",
  "corridorRadius": "5km", // UI 표기용(광역 5km)
  "corridorSummary": "중부고속도로 - 영동고속도로 - 남양로", // UI 표기용(TBT 축약)
  "minFare": 50000,
  "pickupRadius": 5, // 첫짐보다 타이트하게 조임
  "targetRegions": [ // 수동 타겟 도시 무시하고 서버가 내려준 읍면동 배열
    "파장동", "서초동", "반포동", "우면동", "양재동", 
    "신갈동", "정자동", "금곡동" // ... 40~50개
  ]
}
```

## 5. 최종 데이터 파이프라인 (Execution Flow)
1. **(첫짐)** 기사가 앱에서 '첫짐' 수동 필터를 켜고 콜을 하나 낚아채서 확정합니다.
2. **(노선 그리기)** 서버는 카카오내비 API를 치고, 정적인 고정 지도 캔버스(줌 불가)와 함께 `[1: 상차지]`, `[2: 하차지]`를 PinnedRoute에 띄웁니다.
3. **(서버 연산)** 이 주행 노선(Polyline)을 메모리 상의 GeoJSON(`merged_map.geojson`)과 겹쳐서 면적을 계산, 위 `4.2 합짐` JSON을 뚝딱 만들어냅니다.
4. **(필터 교체)** 다음 틱 통신에서 앱은 합짐 JSON을 강제로 받아 자신의 타겟 지역 필터를 오버라이드합니다.
5. **(상태창 변신)** OrderFilterStatus UI는 즉시 보라색 뱃지로 바뀌며 `"합짐 | 광역(5km) | 중부고속 - 신갈JC"` 를 출력합니다.
6. **(자동 인접)** 관제탑에서 GPS가 초속 n M/s 이상으로 움직임이 포착되면, 서버는 은밀하게 `targetRegions`를 5km 버퍼에서 0km(선에만 걸린 동)로 더 깎아서 날려보냅니다. (이게 Auto-Strict 입니다)


> **💡 톨게이트 딜레마 체크됨**: "접해 있어도 고속도로면 국도로 빙 돌아와야 하므로 문제다." ➡️ 나중엔 Turf.js로 연산할 때 도로 타입이 '고속국도'인 구간은 반경을 0으로 만들어버리는 형태로 고도화 가능합니다. 현재는 명확한 합짐 파이프라인 개통을 1순위로 진행하겠습니다.
