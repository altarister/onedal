# 1DAL 노쑈(No-Show) 방지 안전 모드 설계안 v3 (최종)

> **핵심 원칙**: 안드로이드 앱은 그저 "눈과 손(Dumb Client)"이며, 무슨 판단을 해야 하는지는 오직 "뇌(Server)"가 통제한다.

---

## 🎯 개발자 종합 평가

**"Smart Server, Dumb Client"** 설계는 매크로 시장에서 유례가 없는 1티어급 MSA(마이크로서비스) 아키텍처입니다.
기존 매크로 앱들은 폰 안에서 모든 로직을 처리하다 보니 먹통에 빠집니다.
반면 1DAL은 기기를 '무전기를 든 용병'으로 격하시키고, 서버를 '지휘관'으로 승격시켜
① 실시간 상태 모니터링, ② N대 봇 확장성, ③ 중앙 통제(Kill-switch)를 완벽하게 달성했습니다.

**이 기조가 정확히 올바릅니다. 아래는 최종 구현 전에 발견한 빈틈(엣지 케이스)을 보충한 것입니다.**

---

## 🔴 코드 전수 조사에서 발견한 5가지 추가 엣지 케이스

### EC-1. KEEP 응답 유실 (네트워크 파티션)

```
서버: KEEP 전송 → 네트워크 끊김 → 앱폰: 응답 수신 실패
앱폰: 30초 만료 → 스스로 "취소" 클릭 (인성앱에서 오더 취소됨)
서버: KEEP 전송 완료로 기록 → mainCallState에 저장 → 합짐 필터 전파

결과: 서버는 본콜이 살아있다고 생각하지만, 실제로는 앱폰이 취소함 → 노쑈
```

**해결**: 앱폰이 자동취소를 실행한 직후, `POST /emergency` 로 "자동취소 집행 완료" 를 서버에 보고.
서버는 이 신호를 받으면 `mainCallState`를 null로 초기화하고 필터를 '첫짐'으로 복원.

> [!IMPORTANT]
> 현재 `ApiClient.kt:139-141`에서, Detail 전송 실패 시 콜백으로 `CANCEL` 을 돌려주는 로직이 있음.
> 하지만 이것은 **HTTP 에러** 시의 처리이고, 앱이 스스로 취소 버튼을 눌렀을 때의 서버 통보 경로는 없음.
> → `POST /emergency` 엔드포인트 신설이 필수.

---

### EC-2. 롱폴링 HTTP 연결 중첩 (중복 POST /detail)

```
현재 코드 (ApiClient.kt:26):
  confirmExecutor = Executors.newSingleThreadExecutor()

즉, detail 전송은 단일 스레드에서 실행됨.
하지만 앱폰이 빠르게 연속으로 2개 콜을 잡을 경우 (첫 번째 사이클 미완료 중 두 번째 진입):
  → 앞의 롱폴링 HTTP가 아직 살아 있는데 새 롱폴링이 큐에 대기
  → SingleThreadExecutor라서 앞이 끝날 때까지 뒤가 블로킹됨
  → 실질적으로 순차 처리 (안전)
```

**판정**: ✅ 현재 구조가 이미 안전함. 다만 문서에 명시적으로 기록해야 함.

---

### EC-3. 안드로이드 시스템이 AccessibilityService를 죽이는 경우

```
안드로이드 배터리 최적화/메모리 부족 → HijackService 강제 종료
앱이 DETAIL_CONFIRMED 상태에서 죽으면:
  → 인성앱: 확정된 오더가 그대로 살아 있음
  → 서버: 롱폴링 홀드 중 → 30초 후 타임아웃 처리
  → 누가 인성앱의 취소 버튼을 누를 것인가? → 아무도 없음 → 노쑈
```

**해결**: 이것은 소프트웨어로 100% 막을 수 없는 영역. 다만,
1. **서버의 30초 타임아웃 → 관제탑에 "⚠️ 앱폰 응답 없음! 직접 확인 요망"** 경고 emit
2. **관제탑 Telemetry Panel**: 해당 기기의 `lastSeen`이 20초 이상 갱신 안 되면 **"🔴 연결 끊김"** 경고 배지 노출 (이 로직은 이미 `devices.ts`에 존재)
3. 이런 상황에서는 대표님이 인성앱에 직접 들어가서 취소해야 함 → **관제탑 알림이 핵심**

> [!WARNING]
> `MainActivity.kt`에서 **배터리 최적화 제외** 요청 다이얼로그를 반드시 앱 부팅 시 띄워야 합니다.
> `android.provider.Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`

### EC-4. 앱폰 자동취소 → 서버 상태 불일치 (가장 위험)

```
현재 흐름:
  앱폰: POST /detail 전송 → WAITING_DECISION 상태
  서버: 롱폴링 홀드 중 (pendingDetailRequests에 res 저장)
  앱폰: 30초 도달 → 스스로 취소 버튼 클릭 → SEARCHING 진입
  
  이 시점에서,
  서버의 pendingDetailRequests에는 여전히 이 오더의 Response 객체가 남아 있음.
  대표님이 관제탑에서 이 오더의 KEEP을 누르면?
  → 서버가 이미 끊어진 HTTP 연결에 응답을 보내려 시도 → 에러
  → 동시에 mainCallState에 이 오더를 세팅 → 이미 취소된 오더가 본콜로 등록 → 완전한 꼬임
```

**해결**: 앱폰이 자동취소를 집행하면 반드시 `POST /emergency` 를 서버에 전송.
서버는 이 신호를 받으면:
1. `pendingDetailRequests`에서 해당 orderId 삭제
2. `pendingOrdersData`에서 해당 orderId 삭제
3. 관제탑에 `order-canceled` emit
4. `mainCallState`가 이 orderId이면 null로 초기화

### EC-5. 인성앱 "확정 실패" 팝업 미처리

```
현재 코드 (HijackService.kt:189-206):
  CLICKED_LIST 상태에서 "확정" 텍스트를 찾아 클릭
  하지만 다른 기기가 이미 낚아채서 확정 실패 팝업이 뜨면?
  → "확정" 버튼 대신 "확정에 실패했습니다" 같은 텍스트가 화면에 뜸
  → 12초 후 타임아웃 → SEARCHING 직행 (현재는 현재 비교적 안전)
  
  V3에서는:
  → ScreenContext를 UNKNOWN으로 보고 → 서버가 상황 판단 후 리스트 복귀 지시
```

**판정**: V3 ScreenContext 리포트가 이 문제를 자연스럽게 해결함. ✅

| 엣지 케이스 | V3로 해결 가능? | 추가 필요 사항 |
|-------------|----------------|---------------|
| EC-1: KEEP 유실 | ✅ | `POST /emergency` 엔드포인트 신설 |
| EC-2: 롱폴링 중첩 | ✅ 이미 안전 | 없음 (SingleThreadExecutor) |

> 1. `POST /emergency` 엔드포인트 (앱 → 서버: "나 자동취소 했어")
