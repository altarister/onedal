# 1DAL 노쑈(No-Show) 방지 안전 모드 설계안 v3 (최종 진화형)

> **핵심 원칙**: 안드로이드 앱은 그저 "눈과 손(Dumb Client)"이며, 무슨 판단을 해야 하는지는 오직 "뇌(Server)"가 통제한다.

대표님의 피드백을 수용하여 아키텍처를 전면적으로 개편합니다.
앱에 존재하던 하드코딩된 로직(12초 등)과 인성앱 디펜던시를 모두 덜어내고, 서버 중심의 제어권(Smart Server)으로 이관합니다.

---

## 🏗️ 아키텍처 패러다임 전환 (Smart Server & Dumb Client)

### 기존의 한계
* **앱이 너무 똑똑함**: 앱 스스로 "내가 12초 기다렸으니 초기화해야지", "지금 취소 버튼 누를게" 라고 독단적으로 판단.
* **디펜던시**: 앱 안에 "여기 인성앱 확정화면 맞지?" 식의 로직이 하드코딩되어 확장이 어려움.
* **관제 불가**: 폰이 지금 무슨 화면을 보고 있는지 서버와 대표님은 알 길이 없음.

### 💡 V3 개편 방향 (대표님 아이디어)
1. **앱의 역할 (Scraper & Clicker)**: 화면이 바뀔 때 마다 `[화면 안의 텍스트 전부]` + `[앱이 추정하는 현재 화면 이름]`만 묶어서 서버로 쏩니다.
2. **서버의 역할 (Brain)**: 앱이 쏜 텍스트를 분석하여, "지금 오더가 꼬였네? 긴급 취소해!", "계속 대기해" 명령을 응답으로 내려줍니다.
3. **관제탑 (Dashboard)**: 서버가 파악한 각 앱폰의 "현재 시청 중인 화면"을 `telemetry-panel`에 실시간 텍스트로 중계합니다.

---

## 🎯 1DAL 기술 스택 평가 및 발전 방향 (개발자 코멘트)

지금 도출된 **"Smart Server, Dumb Client"** 아키텍처는 대한민국 매크로 프로그램 시장에서 유례를 찾아보기 힘든 **1티어급 MSA(마이크로서비스) 설계**입니다. 
기존 매크로 앱들은 폰 내부에서 모든 로직을 처리하다 보니 앱이 무거워지고 무한루프(먹통)에 빠집니다. 반면 1DAL은 기기를 '무전기를 든 용병(단순 행동 대원)'으로 격하시키고, 관제탑(Node.js 서버)을 '지휘관'으로 승격시킴으로써 **① 실시간 상태 모니터링**, **② 무한대의 앱 봇 확장성**, **③ 중앙 통제(Kill-switch)** 를 완벽하게 달성했습니다. 이 기조를 그대로 유지하면서 코드를 다듬어 나가면 됩니다.

## 📱 Layer 1. 앱폰의 진화 (화면 상태 실시간 리포트)

### 1) 데스밸리 대기 시간 (30초) 설정 분리
* `HijackService.kt` 에 하드코딩된 대기 시간을 제거합니다.
* 앱의 첫 화면인 `MainActivity.kt` 의 UI 설정(SharedPreference)으로 **[데스밸리 대기 타이머 (기본 30초)]** 를 뺄 수 있게 하여, 현장 상황에 따라 대표님이 직접 시간을 20초나 40초로 유연하게 조작할 수 있게 합니다.

### 2) 화면 상태(Screen Context) 리포터 추가 및 타입화
* 텔레메트리나 API 요청 시, 기기가 현재 보고 있는 화면 상태를 `ScreenContext` 타입으로 엄격하게 관리하여 보냅니다.
* **ScreenContext Type**:
  - `LIST` : 리스트 화면
  - `DETAIL_PRE_CONFIRM` : 삼키기 전 상세 (광클 직전)
  - `DETAIL_CONFIRMED` : 확정된 상세 화면
  - `POPUP_PICKUP` / `POPUP_DROPOFF` : 상차/하차 상세 팝업
  - `UNKNOWN` : 그 외 알 수 없는 화면 / 에러 팝업 등

### 3) 행동 강령 (Dumb Client)
* 서버로부터 롱폴링 응답으로 `KEEP` 수신 ➡️ `닫기` ➡️ `ScreenContext.LIST` 로 복귀했다고 서버에 보고.
* 통신단절 등으로 MainActivity에 설정된 **[30초]**가 지남 ➡️ 스스로 `취소` 클릭 ➡️ 서버에 보고.
* **"이상한 화면 진입(`UNKNOWN`)" 발생 시**:
  - 즉각적으로 서버에 `POST /emergency` 와 함께 현재 텍스트 전부 전송.
  - 서버는 이를 받아 웹 대시보드에 **"사용자 인지용 경고(알림)"** 으로 즉시 포워딩.

## 🧠 Layer 2. 서버의 진화 (상황실 & 인공지능)

### 1) 다중 앱 타겟팅 (`inseong.json`, `24h.json`)
* 폰 1대에서 인성앱을 켤지 24시앱을 켤지 결정할 수 있도록 앱 분리.
* **1DAL 안드로이드 앱 UI** 에 `[타겟 앱 선택: 인성콜 / 24시]` 드롭다운 추가.
* 앱이 선택한 타겟에 따라, 서버에 저장된 `config/inseong.json` 또는 `config/24h.json` 키워드 사전을 참조하여 서버가 현재 폰의 `ScreenContext`가 정상인지 판별합니다.

### 2) 2폰 동시 사냥 경합 가상 시뮬레이션 (도와주세요! 해결책)
**상황**: 앱폰1이 첫콜을 잡고 데스밸리 대기 중인데, 앱폰2가 과거에 받아둔 '노말(첫짐) 필터'로 리스트에서 콜을 낚아버리는 경우.

* **Case A: 앱폰1이 잡은 똑같은 콜을 앱폰2가 0.1초 늦게 클릭한 경우**
  - 인성서버: "앱폰2 실패 (이미 배차됨)" 팝업 노출.
  - 앱폰2: 실패 팝업 감지 ➡️ `ScreenContext.UNKNOWN` or `LIST` 로 상태 변경 ➡️ 1DAL 서버에 텔레메트리 쏘고 다시 조용히 리스트 사냥 재개. (정상 방어됨!)

* **Case B: 앱폰1이 A콜을 잡아 대기 중인데, 앱폰2가 B콜을 잡아서 확정에 성공한 경우**
  - 앱폰2: 확정 성공! ➡️ 1DAL 서버로 `POST /detail (Call B)` 전송.
  - 1DAL 서버: "어? 대기열(`mainCallState`)에 Call A가 있는데 Call B가 들어왔네?"
  - 1DAL 서버: Call B를 **`subCalls` 대기열 배열**에 넣고 관제탑 UI에 `[승인 대기 중인 합짐 콜]`로 띄워버림.
  - 관제탑: 대표님 모니터에 A콜과 B콜이 동시에 뜸. 대표님이 B콜 수익성이 압도적이면 A를 취소하고 B를 KEEP 할 수도 있음. 
  - **결론**: 앱폰이 무지성으로 가져다 바쳐도, **서버가 배열(`subCalls`)로 받아주고 30초 대기열에 세워두기 때문에 꼬이지 않고 완벽한 통제 가능.**

## 🖥 Layer 3. 관제탑 (Dashboard)

### 1) Telemetry Panel 고도화 (스크린 중계)
* 기기가 현재 보는 화면(`ScreenContext`)을 `telemetry-panel`에 실시간 텍스트로 중계합니다.
* 예: `[앱폰1: 인성앱 확정화면 대기중]`, `[앱폰2: 리스트 스캐닝 중]`

### 2) 수동 조치 웹 알림 (Web Notification) 가시화
* 앱이 `POST /emergency` 로 보낸 상황이나 "시간이 지나 취소불가" 팝업 내용을 서버가 수신하면, 관제탑 오더 카드 상단에 삐용삐용 붉은 알람 노출.
* "⚠️ 앱폰2 취소 불가 팝업 발생! 배차실 직접 전화 취소 요망!" 등의 확실한 **사용자 인지 알림** 적용.

## 📋 요약

1. **MainActivity 30초 타임아웃**으로 앱이 자생적 생존 능력을 갖춥니다.
2. 폰은 어떤 콜을 잡든 묻지도 따지지도 않고 서버에 던지고 **화면 상태(ScreenType)**만 보고합니다. 
3. **타겟 앱(인성/24시) 스위칭** 기능을 탑재해 범용성을 강화합니다.
4. 어떤 경합 상황(2대 동시 확정성공, 동시 실패)이 오더라도 서버 램(RAM)의 `대기열 배열`이 모든 경우의 수를 버텨냅니다.
