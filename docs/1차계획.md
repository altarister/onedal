# 1DAL 노쑈(No-Show) 방지 안전 모드 설계안

> **원칙**: 이전 오더 사이클이 완전히 끝날 때까지 새 오더를 절대 잡지 않는다.

---

## 🔴 현재 구조에서 오작동이 발생하는 3가지 핵심 원인

### 원인 1. 앱폰 타임아웃 → SEARCHING 강제 롤백 중 화면은 여전히 인성앱 상세 페이지

```
현재 코드 (HijackService.kt:96-103):
  12초 경과 → currentState = SEARCHING, currentOrder = null

실제 상황:
  앱폰은 인성앱 "확정 후 상세 페이지"에 있음
  상태는 SEARCHING이 됐으나 화면은 확정이 된 상태
  → 다음 리스트에서 새 오더를 잡으려 클릭 → but 위 화면이 열려있음
  → "취소" 버튼도 안 누름 → 인성앱 기준으로 이미 확정된 오더를 "방치"
  → 노쑈 발생
```

### 원인 2. 롱폴링 30초 타임아웃 → 앱폰에 CANCEL 자동 전송 → 버튼 클릭 실패

```
현재 코드 (detail.ts:206-217):
  30초 경과 → heldRes.json({ action: 'CANCEL' })
  앱폰: EXECUTING_DECISION 진입 → "취소" 버튼 찾아 클릭

문제:
  30초 후 화면이 이미 리스트로 돌아가 있으면 "취소" 버튼 없음
  → 클릭 실패 → 12초 후 타임아웃 → 강제 SEARCHING
  → 인성앱엔 확정된 오더가 살아있음 → 노쑈
```

### 원인 3. 앱폰 2대 동시 확정 경합 — 합짐 오더의 자동 패스

```
현재 코드 (detail.ts:53-65):
  KEEP → mainCallState가 없으면 본콜, 있으면 subCalls.push()

문제:
  앱폰2가 합짐을 낚아채 인성앱에서 확정 완료
  → 서버/관제탑 검토 없이 이미 확정된 상태
  → 잘못된 합짐이면 노쑈 위험
```

---

## 🎯 설계 원칙: 단순하게, 안전하게

> **"1개 오더 사이클이 완전히 종결(인성앱에서 닫기 or 취소 완료)될 때까지 새 오더를 절대 잡지 않는다"**

이를 위해 **글로벌 잠금(Global Lock)** 과 **비상 복구 상태(EMERGENCY_RECOVERY)** 를 도입합니다.

---

## 🔒 안전 모드 설계: 3개 레이어 변경

### Layer 1: 앱폰 (HijackService.kt) — 글로벌 잠금 + 비상 복구 상태 추가

**새 상태 추가**: `EMERGENCY_RECOVERY`

| 현재 | 개선 |
|------|------|
| 타임아웃 12초 → SEARCHING 강제 롤백 | 타임아웃 시 → **EMERGENCY_RECOVERY** 진입 |
| EXECUTING_DECISION 버튼 못 찾으면 재시도 | 30초 이상 버튼 못 찾으면 → **EMERGENCY_RECOVERY** |
| SEARCHING으로 가는 경로 여러 곳 | **SEARCHING으로 가는 유일한 경로**: EXECUTING_DECISION 성공 or EMERGENCY_RECOVERY 복구 완료 |

**`EMERGENCY_RECOVERY` 동작 절차**:
```
1. 현재 화면에 "취소" 버튼이 있으면 클릭 (인성앱 확정 취소 시도)
2. "취소" 버튼 없으면 → 백 버튼으로 인성앱 리스트 강제 복귀
3. 리스트 화면 진입 텍스트("요금" 등) 확인 후 → SEARCHING 진입
4. 서버에 EMERGENCY 이벤트 POST 전송 (서버 메모리 초기화 요청)
```

---

### Layer 2: 서버 (detail.ts) — 타임아웃 자동 CANCEL 제거

**왜 타임아웃 CANCEL 자동 전송이 위험한가**:
```
타임아웃 시점에 앱폰의 화면 상태를 서버는 알 수 없음
  - 인성앱이 이미 다른 화면이면 "취소" 버튼 없음
  - CANCEL 받은 앱폰: 버튼 못 찾음 → 12초 후 타임아웃 → SEARCHING
  - 인성앱: 확정된 오더 방치 상태 = 노쑈
```

**개선된 타임아웃 흐름**:

| 현재 | 개선 |
|------|------|
| 30초 후 → CANCEL 자동 전송 | 30초 후 → 관제탑에 **"⚠️ 응답기한 초과"** 경고만 emit |
| 앱폰이 알아서 취소 | 관제탑에서 **KEEP / CANCEL / EMERGENCY** 수동 선택 |
| EMERGENCY 액션 없음 | EMERGENCY 선택 시 → 앱폰에 비상 복구 명령 전송 |

---

### Layer 3: 관제탑 (Dashboard) — 상태 가시화 + 비상 버튼

**변경 방향**:

| 현재 | 개선 |
|------|------|
| KEEP/CANCEL 버튼만 있음 | + **"🚨 비상 취소"** 버튼 항상 표시 |
| 타임아웃 → 조용히 자동 처리 | 타임아웃 → 붉은 경고 배너 + 수동 개입 요구 |
| 오더 종료 시 카드 바로 삭제 | 앱폰 SEARCHING 복귀 확인 후 카드 삭제 |

---

## ✅ 개선 후 정상 흐름

```
[앱폰] 꿀콜 감지 → 광클 → 확정
        ↓ (SEARCHING 잠금)
[앱폰] 출발지/도착지/적요 스크래핑 → 서버 전송
        ↓
[서버] 카카오 연산 → 관제탑에 결과 emit
        ↓
[관제탑] 대표님이 KEEP / CANCEL 선택
        ↓
[서버] 앱폰에 판결 전달
        ↓
[앱폰] EXECUTING_DECISION → 버튼 클릭 성공
        ↓
[앱폰] SEARCHING 잠금 해제 (새 오더 가능)
```

**비상 상황 흐름**:
```
중간에 이상 (타임아웃 or 버튼 못찾음)
        ↓
[앱폰] EMERGENCY_RECOVERY → "취소" 버튼 클릭 시도
        ↓ 실패 시
[앱폰] 백 버튼 → 리스트 화면 복귀 확인
        ↓
[앱폰] 서버에 EMERGENCY 신호 전송
        ↓
[서버] 해당 오더의 메모리 초기화
        ↓
[앱폰] SEARCHING 진입
```

## ⚠️ 결정 필요한 열린 질문

> [!IMPORTANT]
> 구현 전 아래 3가지를 확인해 주세요.

1. **비상 복구 시 취소 불가 케이스**: 인성앱에서 이미 배차 진행 중이라 "취소" 버튼 자체가 없는 상황이 있나요? 이 경우 그냥 리스트로만 돌아가도 되나요?

2. **앱폰 운용 대수**: 안전 모드 안정화 전까지 **앱폰 1대만** 쓰실 건가요, 아니면 2대(본콜+합짐) 구조를 유지하실 건가요?

3. **타임아웃 시간**: 관제탑에서 KEEP/CANCEL 판단하시는 데 보통 몇 초 걸리세요? 30초가 충분한지 확인 필요합니다.
