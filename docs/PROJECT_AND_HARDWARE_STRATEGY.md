## 1. 프로젝트(코드) 구조 전략

### 추천 구조: Polyrepo (역할 완전 분리)
```text
/Users/seungwookkim/reps/onedal/
├── onedal-app/  ← 현재 1dal 폴더 이름 변경
│   └── 안드로이드 스튜디오 (Kotlin)
│       역할: 화면 감지 → 터치 → 서버로 HTTP POST
│
└── onedal-web/  ← 새로 생성
    └── Next.js (React + Node.js API Routes)
        역할: 콜 수신 API + 대시보드 UI + 소켓 알림
```

**왜 분리하는가?**
- 안드로이드(Gradle/Kotlin)와 웹(npm/TypeScript)은 빌드 시스템이 완전히 다릅니다.
- 한 폴더에 섞으면 IDE 자동완성이 꼬이고, 빌드가 충돌합니다.
- 서버 코드는 `onedal-web`의 `src/app/api/` 안에 API Routes로 함께 둡니다.
  (Next.js를 쓰면 프론트엔드와 백엔드를 한 프로젝트에서 관리 가능)

**대시보드는 웹(크롬 브라우저)으로 만드는 것이 정답입니다.**
안드로이드 앱을 또 만들 필요가 없습니다. 프론트엔드 기술(React)로 화면을 만들고, 관제용 폰에서 **크롬 브라우저로 접속**하여 전체 화면 모드로 띄워두면 네이티브 앱과 100% 동일하게 작동합니다.

## 3. 모바일 웹 소켓(WebSocket) 안정성 대책

> **핵심 우려:** 관제용 폰(1호기)에서 크롬 웹으로 소켓을 받는데, 화면이 꺼지거나 다른 앱으로 전환하면 소켓이 끊기지 않을까?

### 대책 1: Screen Wake Lock API (화면 꺼짐 방지)
```javascript
// 대시보드 웹페이지에 이 한 줄만 추가
navigator.wakeLock.request('screen');
```
- 이 코드가 삽입된 웹페이지가 열려 있는 동안은 폰 화면이 절대 꺼지지 않습니다.
- 카카오내비처럼 "항상 화면 ON" 상태가 되어 소켓이 끊길 일이 없습니다.
- 어차피 운전 중에는 거치대에 폰을 두고 계속 보므로, 화면을 끌 이유가 없습니다.

### 대책 2: 자동 재연결 (Auto-Reconnect)
```javascript
// Socket.io는 기본 내장, 순수 WebSocket이면 아래 패턴 적용
socket.onclose = () => {
  setTimeout(() => connectSocket(), 3000); // 3초 후 자동 재접속
};
```
- 만에 하나 네트워크가 갈아타거나(Wi-Fi → LTE) 잠깐 끊겨도, 3초 이내에 자동으로 다시 붙습니다.
- Socket.io 라이브러리를 쓰면 이 기능이 기본 내장되어 있어서 코드 한 줄도 필요 없습니다.

### 대책 3: [추후] 하이브리드 웹뷰 앱 (최종 진화)
- 나중에 "크롬 주소창이 보이는 게 싫다, 진짜 앱처럼 만들고 싶다" 하시면?
- 안드로이드 스튜디오에서 `WebView`라는 껍데기를 만들고(코드 10줄), 승욱님의 웹 URL 주소를 그 안에 넣으면 **네이티브 앱과 구분이 불가능한 하이브리드 앱**이 10분 만에 탄생합니다.
- 이렇게 하면 Foreground Service로 등록이 가능해져 소켓이 절대 죽지 않는 무적 상태가 됩니다.

## 4. 전문가 검토 (Review & Alternatives)

### ✅ 현재 제안의 실현 가능성 평가
| 항목 | 판정 | 근거 |
|------|------|------|
| Polyrepo 분리 | ✅ 실현 가능 | 안드로이드와 웹의 빌드 시스템이 완전히 달라 분리가 자연스러움 |
| 웹 대시보드 (크롬) | ✅ 실현 가능 | Wake Lock + Socket.io로 네이티브 앱 수준의 안정성 확보 가능 |
| 갤럭시 2대 분리 운용 | ✅ 실현 가능 | 현직 기사님들의 표준 세팅과 동일. 현장 검증 완료 |
| Next.js API Routes | ✅ 실현 가능 | 프론트엔드 개발자가 별도 백엔드 없이 서버를 운영하는 가장 효율적 방법 |

### 🤔 검토한 대안들 (왜 채택하지 않았는가?)
| 대안 | 불채택 이유 |
|------|------------|
| Flutter로 앱 2개 통합 | AccessibilityService를 쓸 수 없음. 스캐너 기능 구현 불가 |
| React Native 대시보드 앱 | 웹으로 충분한데 굳이 네이티브 빌드 복잡성을 추가할 이유 없음 |
| Firebase Cloud Messaging (푸시) | 소켓 대신 푸시만 쓰면 실시간성이 1~3초 지연됨. 소켓이 우월 |
| 서버를 별도 레포로 분리 | 1인 개발에서 3개 레포 관리는 과잉. Next.js API Routes로 2개면 충분 |
| Monorepo (Turborepo 등) | Gradle + npm 혼합 Monorepo는 설정 지옥. 1인 개발자에겐 독 |

### 💡 추가 제안 (더 좋아질 수 있는 점)
1. **onedal-web 배포**: 노트북을 24시간 켜두지 않으려면, Vercel이나 Railway 같은 무료/저가 클라우드에 올려두시면 됩니다. Next.js는 Vercel과 찰떡궁합이라 `git push`만 하면 자동 배포됩니다.
2. **스캐너 폰 전원 관리**: 차량 시거잭 USB 충전기로 항시 충전 상태를 유지하세요. 접근성 서비스(onedal-app)는 배터리를 꽤 먹습니다.
3. **DB 선택**: MVP 단계에서는 SQLite(파일 DB)나 Vercel KV로 충분하고, 나중에 데이터가 쌓이면 Supabase(무료 PostgreSQL)로 갈아타시면 됩니다.
