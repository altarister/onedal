# 1DAL Platform: 스마트 회랑(동적 합짐) 필터 엔진 최종 마스터 기획서

그동안 논의했던 기획서 [합짐계획1 ~ 합짐계획4]의 모든 피드백과 통찰(톨게이트 딜레마, 다형성 UI, 오버로딩 기법, TS 인터페이스 표준화)을 한 줄의 누락 없이 통합한 **최종 결론 아키텍처 문서**입니다. 향후 개발의 "바이블"이 됩니다.

---

## 1. 아키텍처 철학 (Architecture Philosophy)
새로 도입되는 "스마트 합짐 모드"는 시스템 성능 하락 없이 유연성을 극대화해야 합니다.
1. **Server-Heavy, Client-Light**: 모든 공간 연산(Turf.js)과 경로 추출은 서버 백엔드가 전담합니다.
2. **분기 없는 다형성 (Polymorphism)**: 클라이언트 프론트엔드는 "합짐인지, 첫짐인지"를 따지지 않고, 오직 서버가 던져준 텍스트 덩어리를 매핑할 뿐입니다.
3. **API 비용 제로화**: 합짐 영역을 찾기 위해 카카오 API 역지오코딩을 쏘지 않고, 로컬의 `merged_map.geojson`을 서버 메모리에 올려 최단시간 내에 교차점을 찾습니다.

## 2. 용어 정의 및 TS 타입 (Terminology & Types)
상태값(`status`)을 폐기하고, 앱과 서버 간 결합도를 낮추는 명확한 `boolean` 스위치를 도입했습니다.

### 2-A. 필터 타입 (`shared/src/index.ts`)
```typescript
// 자동배차 설정 인터페이스 (전역 설정 동기화용)
export interface AutoDispatchFilter {
    model: '1t' | '다마스' | '라보' | '오토바이';  // 차량 모델 
    isActive: boolean;                            // 필터링(매크로) 활성화 여부
    isSharedMode: boolean;                        // 🌟 핵심 스위치: 첫짐/합짐 분기
    pickupRadiusKm: number;                       // 상차지 탐색 반경(km)    
    minFare: number;                              // 최소 운임 하한선
    maxFare: number;                              // 최대 운임
    destinationCity: string;                      // UI 표기용 오버로딩 필드 (다형성 구현)
    destinationRadiusKm: number;                  // UI 표기용 반경 필드
    excludedKeywords: string;                     // 제외 단어
    destinationKeywords: string;                  // (내부숨김) 앱 파싱용 50개 읍면동 리스트
    customFilters: string[];                      // 하단 빠른 설정 텍스트
}
```

### 2-B. 경로(Route) 타입 (`PinnedRoute.tsx` 전용)
클라이언트 화면 랜더링에 쓰일 정적 캔버스와 요약 정보 데이터 모델입니다.
```typescript
export interface CorridorRouteData {
    summaryText: string;
    totalDistanceKm: number;
    totalTimeMinutes: number;
    tollFare?: number;
    waypoints: { lat: number; lng: number; type: 'PICKUP' | 'DROPOFF'; label: string; }[];
    alternatives?: { id: string; name: string; timeMinutes: number; distanceKm: number; }[];
}
```

## 3. 동적 UI 매핑 (Polymorphic Overloading)
프론트엔드(`OrderFilterStatus.tsx`) 소스코드에는 어떠한 if문 조건이나 스타일 분기 로직도 들어가지 않습니다. 무조건 아래 순서대로 화면에 뿌립니다.
> `[isSharedMode] | [minFare] | [pickupRadiusKm] | [destinationCity]([destinationRadiusKm]) | [model]`

서버가 상황에 따라 데이터를 주무르면 프론트엔드 UI는 알아서 옷을 갈아입습니다.

**[ 매핑 시나리오 1: 첫짐 세팅 ]**
- 서버 페이로드: `isSharedMode`: false, `minFare`: 4.0, `pickupRadiusKm`: 30, `destinationCity`: "광주시", `destinationRadiusKm`: 10
- 화면 출력 ➡️ **`첫짐 | 4.0 | 30km | 광주시(10km) | 다마스`**

**[ 매핑 시나리오 2: 합짐 세팅 ]** *(광역 10km 버퍼 기준, 현재 반경 0km 진입)*
- 서버 페이로드: `isSharedMode`: true, `minFare`: 4.0, `pickupRadiusKm`: 5, `destinationCity`: "10km확장 | 영동고속 - 남양로 - 경충대로", `destinationRadiusKm`: 0
- 화면 출력 ➡️ **`합짐 | 4.0 | 5km | 10km확장 | 영동고속 - 남양로 - 경충대로(0km) | 다마스`**

## 4. 코어 배차 파이프라인 (Execution Flow)

| 단계 | 주체 | 동작 핵심 내용 |
| :--- | :--- | :--- |
| **1. 낚시** | 앱 | 빈차 상태에서 `첫짐 필터(수동)` 레이더를 가동하여 콜을 탐색합니다. |
| **2. 확정** | 앱/웹 | 기둥콜(본콜)을 잡으면, 웹엔 줌(Zoom) 기능이 제한된 깔끔한 단일 지도 캔버스가 그려지고 마커가 꽂힙니다. 카카오 대안 경로 2~3개가 하단에 노출되고 기사가 하나를 선택해 고정합니다. |
| **3. 연산** | 서버 | 카카오 노선을 바탕으로 Turf.js를 가동, `merged_map.geojson`을 스캔하여 경로 주변 읍/면/동 50여 개를 도출합니다. 이 목록을 `destinationKeywords` 안에 숨겨 담아냅니다. |
| **4. 주입** | 서버 | 다음 `/api/scrap` 하트비트 시, 서버는 방금 만든 JSON(`isSharedMode: true`)을 응답으로 몰래 내려줍니다. |
| **5. 강제덮어쓰기** | 앱 | 앱은 이 데이터를 받아 기존 필터를 전부 폭파시키고, 서버판 '합짐 필터'로 100% 덮어쓰기 합니다. 이제 앱은 내가 지나갈 50개의 동네에서 뜨는 꿀콜만 스크래핑합니다. |

## 5. 심화 방어 로직 (Deep Dive Defenses)

### 5.1 톨게이트 우회 방지 (Highway Detour Penalty)
직선상으로 가까워도 고속도로면 국도를 빙 돌아야 하는 물류의 치명적 딜레마를 서버 연산으로 방어합니다.
* 카카오 API `guides`에서 해당 구간이 `고속국도`/`도시고속도로` 판정을 받으면 Turf.js 합짐 반경 연산을 0km로 제재하여 고속도로를 헤매는 불상사를 원천 차단합니다.

### 5.2 위치 기반 오토 스위칭 (GPS Auto-Strict)
차량이 화물을 싣고 등속으로 움직이기 시작하여 **실주행** 상태임이 파악되면, 서버는 어떠한 조작 없이도 강제로 `destinationRadiusKm`을 `0`(정밀 노선)으로 깎은 필터를 앱에 주입시킵니다. 주행 이탈 확률이 소멸됩니다.
