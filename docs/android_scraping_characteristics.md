# 안드로이드 접근성(Accessibility) 기반 웹/앱 스크래핑 특성 문헌

이 문서는 1DAL 매크로 시스템 고도화 과정 중, 안드로이드 접근성 엔진(`AccessibilityService`)이 화면의 텍스트를 인식하고 처리하는 고유의 패턴과 한계점, 그리고 대응 방안을 기록한 자료입니다. 향후 새로운 배차 앱(원콜, 24시 등)을 연동하거나 UI 업데이트에 대응할 때 반드시 참고해야 합니다.

## 1. 텍스트 병합(Squashing) 현상: 웹 화면 vs 네이티브 화면

### ❌ 웹뷰(Chrome) 기반 화면의 텍스트 압축 증후군
*   **현상**: 브라우저는 HTML 요소(`<div>`, `<span>`) 사이에 명시적인 여백 문자(`&nbsp;` 등)가 없으면, 화면 상에는 띄어쓰기가 되어 보이더라도 접근성 엔진에게는 **모든 글자를 하나로 뭉개서 단일 노드로 전달**합니다.
*   **예시**: 화면에는 `6.8` `경기 광명시` `서울 종로구` 로 보이지만, 안드로이드 로그에는 `6.8경기 광명시-서울 종로구` 즉, `6.8`과 `경기` 사이의 공백이 말살되어 들어옵니다.
*   **원인**: 브라우저의 DOM Tree를 접근성 트리로 변환하는 브릿지의 특성 때문입니다.

### ✅ 네이티브(Android Native) 앱의 독립적 렌더링
*   **현상**: 진짜 배차 앱 (안드로이드 순수 개발 앱)은 `TextView`와 같은 개별 레이아웃 상자에 글자를 담습니다.
*   **예시**: 네이티브 앱은 `6.8`, `30.9`, `서울 종로구` 가 각각 독립된 텍스트 블록(`ScreenTextNode`)으로 분리되어 아주 깨끗하게 수집됩니다.
*   **해결책(웹 대응)**: 웹 화면의 오염된 텍스트(`6.730.9서울`)를 처리하기 위해, 단순 문자열 매칭이 아닌 **의미론적 정규식(사금 채취식 파싱)**을 도입하여 내부 속살(숫자와 주소)만 핀셋으로 뽑아내는 방어를 적용해야 모든 종류의 앱에서 범용 파싱이 가능합니다.

### 🔑 핵심 발견: `<span>` vs `<div>` 태그가 분리 여부를 결정한다
*   **발견일**: 2026-04-09
*   **현상**: 웹 화면의 HTML에서 `<span>` (인라인 요소)을 `<div>` (블록 요소)로 바꾸자, 안드로이드 접근성 엔진이 각 항목을 독립된 텍스트 노드로 완벽하게 분리해서 인식하기 시작했다.
*   **원인**: 크롬 브라우저의 접근성 트리 변환기는 인라인(inline) 요소들을 하나의 텍스트 노드로 합치고, 블록(block) 요소들만 각각 독립된 노드로 생성한다.

| HTML 태그 | CSS 특성 | 접근성 엔진 인식 |
|-----------|---------|----------------|
| `<span>` | inline (한 줄에 나란히) | 옆 요소들과 **합쳐서** 단일 노드로 전달 → 텍스트 짬뽕 |
| `<div>` | block (줄바꿈) | 각각 **독립된** 노드로 전달 → 완벽 분리 |

*   **결론**: 웹 목업 테스트 화면을 만들 때 배차 리스트의 각 셀(거리, 주소, 요금 등)을 `<div>`로 감싸면, 사금 채취식 정규식 없이도 네이티브 앱과 동일한 수준의 텍스트 분리가 가능하다.

## 2. 화면 스크롤과 노드 소멸 (Viewport Culling) 메모리 관리 현상

### 🔍 스크롤에 따른 텍스트 생존 여부 (안드로이드의 최적화)
*   **잘못된 가설**: "안드로이드 엔진은 스크롤 맨 밑의 안 보이는 글씨까지 한 덩어리로 인식해 다 긁어올 것이다." (틀림)
*   **검증된 진실**: 안드로이드 운영체제(특히 Compose의 `LazyColumn`이나 기존 `RecyclerView`)는 메모리 폭발을 막기 위해, 사용자가 스크롤을 내려 **화면 밖으로 밀려난 시각 요소는 접근성 트리에서 즉시 파기(삭제)해 버립니다.** 반대로 아래쪽에서 새롭게 나타나는 글자는 그때 비로소 트리 구조에 새롭게 생성됩니다.
*   **예외**: `MainActivity`의 통계 화면처럼 개발자가 애초에 아주 긴 줄바꿈(`\n`)이 포함된 '단 하나의 텍스트 상자'를 쑤셔넣은 경우에만 한 덩어리로 취급됩니다.

### 📜 리스트 전체(스크롤 이면) 스크래핑 구현 원칙
리스트 밑바닥에 숨겨진 오더들까지 전부 가져오려면 **자동 스크롤 및 지능형 병합(Chain Scraping)** 프로세스가 필수적입니다.
1. 현재 화면의 모든 텍스트를 스크랩 및 파싱.
2. 매크로가 타겟 리스트 노드에 접근성 스크롤 명령 발사: `performAction(AccessibilityNodeInfo.ACTION_SCROLL_FORWARD)`
3. 화면 변경 이벤트(`TYPE_WINDOW_CONTENT_CHANGED` 또는 `TYPE_VIEW_SCROLLED`) 감지.
4. 기존에 스크랩한 콜 리스트의 마지막 항목과 새롭게 등장한 항목을 비교(ID, 시간, 주소 해시 등)하여 중복을 제거하고 병합.
5. 리스트 끝에 도달할 때까지 위 과정을 매우 빠른 속도로 반복.

## 3. 오버 스크래핑 (Over-Scraping): 눈에 보이지 않는 텍스트 수집

접근성 엔진은 기기 화면에 존재하는 '글꼴 속성'을 가진 모든 객체를 폭식합니다.
*   **웹페이지의 문서 제목 (Title)**: 스크린 상단 웹 브라우저 탭에 숨겨져 있는 웹 페이지의 원래 제목 (예: `Sign in - Google Accounts`)이 수집됩니다.
*   **운영체제 시스템 UI (System UI)**: 멀티태스킹 메뉴 하단의 버튼("모두 지우기", "스크린샷"), 상태 표시줄 등 배차 화면 외부의 기기 고유 시스템 텍스트까지 무차별적으로 긁어옵니다.
*   **방어 전략**: 파싱 엔진은 긁어온 텍스트 무더기에서 배차 앱 고유의 시그니처 텍스트("요금", "거리", "상차지")를 찾고, 배차 박스의 기하학적 세로/가로축(Y, X 좌표) 안에 온전히 포함된 노드들만 그룹핑(Grouping)해내는 철저한 정제 필터가 필요합니다.
