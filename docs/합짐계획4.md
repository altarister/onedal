# 1DAL Platform: 스마트 회랑(동적 합짐) 최종 기획 및 아키텍처

대표님과 논의하며 나온 가장 우아한 인사이트(다형성 UI 컴포넌트, `model` 속성 추가)를 최종 단계로 집대성했습니다.

## 1. 용어 정의 및 TS 타입 (Terminology & Types)

상태값(`status`)을 버리고, `boolean`(`isActive`, `isSharedMode`) 기반으로 리팩토링합니다. 추가로 차량 제원 필터(`model`)가 추가되었습니다.

### 1-A. 필터 타입 (`shared/src/index.ts`)
대표님 제안대로 속성을 재정의합니다. 
(※ 주석: 앱이 50여 개의 합짐 읍/면/동 텍스트를 파싱해야 하므로 내부 데이터 저장용간인 `destinationKeywords`는 보이지 않는 필드로 유지하여 앱 파싱 로직을 손대지 않게 했습니다.)

```typescript
// 자동배차 설정 인터페이스 (전역 설정 동기화용)
export interface AutoDispatchFilter {
    model: '1t' | '다마스' | '라보' | '오토바이';  // 차량 모델 (예: "1톤카고", "다마스")
    isActive: boolean;                            // 필터링(매크로) 활성화 여부
    isSharedMode: boolean;                        // 첫짐/합짐 분기 (true면 합짐 회랑, false면 첫짐 수동)
    pickupRadiusKm: number;                       // 내위치 반경 상차지 탐색(km)    
    minFare: number;                              // 최소 운임 (하한선)
    maxFare: number;                              // 최대 운임 (디폴트 100만)
    destinationCity: string;                      // 하차 목표 메인 지역 (시/군/자치구) / 합짐 시에는 UI 축약 문구로 오버로딩
    destinationRadiusKm: number;                  // 하차 목표 주위 탐색 반경 (km)
    excludedKeywords: string;                     // 제외 단어 (콤마 분리 문자열)
    destinationKeywords: string;                  // (내부망) 앱 파싱용 읍/면/동 50개 콤마 분리 문자열
    customFilters: string[];                      // 특수 기호 등 하단 빠른 설정 텍스트 (ex: "^^,@", "김포,인천...")
}
```

### 1-B. 경로(Route) 타입 (`PinnedRoute.tsx` 전용)
경로(Route) 데이터는 필터와 완전히 분리되어야 합니다. 카카오 API 데이터를 정제하여 클라이언트 뷰어에 최적화합니다.
```typescript
export interface CorridorRouteData {
    summaryText: string;
    totalDistanceKm: number;
    totalTimeMinutes: number;
    tollFare?: number;
    waypoints: { lat: number; lng: number; type: 'PICKUP' | 'DROPOFF'; label: string; }[];
    alternatives?: { id: string; name: string; timeMinutes: number; distanceKm: number; }[];
}
```

## 2. 동적 UI 매핑 설계 (Polymorphic UI Component)
> **대표님 핵심 설계**: 동일한 코드값(`OrderFilterStatus.tsx`의 컴포넌트 렌더링 코드 1벌)만으로 **첫짐/합짐**을 모두 표현하는 아키텍처.

프론트엔드(`OrderFilterStatus.tsx`)는 아무런 로직 분기 없이 단순히 아래 데이터 맵핑만 렌더링합니다.
`[isSharedMode] | [minFare] | [pickupRadiusKm] | [destinationCity]([destinationRadiusKm]) | [model]`

서버가 상황에 맞게 데이터를 세팅해 주면 프론트엔드는 마법처럼 각기 다른 상태를 예쁘게 띄우게 됩니다.

**[ 매핑 시나리오 1: 첫짐 세팅 ]**
- 서버 페이로드: `isSharedMode`: false, `minFare`: 4.0, `pickupRadiusKm`: 30, `destinationCity`: "광주시", `destinationRadiusKm`: 10
- 화면 출력 ➡️ **`첫짐 | 4.0 | 30km | 광주시(10km) | 다마스`**

**[ 매핑 시나리오 2: 합짐 세팅 (광역 10km 버퍼, 현재 0km 정밀 주행중) ]**
- 서버 페이로드: `isSharedMode`: true, `minFare`: 4.0, `pickupRadiusKm`: 5, `destinationCity`: "10km확장 | 영동고속 - 남양로 - 경충대로", `destinationRadiusKm`: 0
- 화면 출력 ➡️ **`합짐 | 4.0 | 5km | 10km확장 | 영동고속 - 남양로 - 경충대로(0km) | 다마스`**

이 구조는 프론트엔드 코드(if-else 분기)를 극단적으로 제거하면서도, 백엔드에서 텍스트의 자유도를 완전히 가져갈 수 있는 **역대급 다형성 설계(Polymorphism)**입니다!

## 3. 코어 파이프라인 (Execution Flow)
1. **[빈차 상태]**: 앱은 수동 세팅된 '첫짐 필터'로 낚시질을 합니다. (`isSharedMode: false`)
2. **[본콜 확정]**: 본콜(기둥콜)을 잡는 순간, UI에는 줌(Zoom) 기능 없는 깔끔한 **단일 정적 캔버스(PinnedRoute)** 에 `1: 상차지`, `2: 하차지` 핀이 꽂힙니다.
3. **[서버 회랑 계산]**: 서버는 카카오 경로를 받아 Turf.js로 노선을 따고 50개의 해당 읍면동 배열을 `destinationKeywords` 문자열 안에 숨겨 담아냅니다.
4. **[S2S 필터 주입]**: 서버는 `[매핑 시나리오 2]` 처럼 조합한 JSON을 `/api/scrap` 응답으로 앱에 내려보냅니다.
5. **[앱 자동전환]**: 앱은 이를 덮어쓰고, 프론트엔드의 대시보드는 즉각 보라색 뱃지로 전환되며 합짐 모드를 알립니다.

## 4. 심화 고민: 톨게이트 딜레마 / GPS 스위칭
*   **고속도로 톨게이트 딜레마**: 카카오 가이드에서 `고속국도`/`도시고속도로` 구간은 Turf.js 폴리곤 생성 시 반경을 0km로 제재하여 고속도로를 빠져나와 헤매는 일을 방지합니다.
*   **GPS 속도 기반 Auto-Strict**: 기사가 운전을 시작하여 실주행 속도가 감지되면 서버는 즉시 `destinationRadiusKm` 값을 강제로 `0`(정밀 노선)으로 깎아 폰에 다시 덮어씌웁니다. 주행 이탈 확률을 없앱니다.
