# 📱 앱 콜 데이터 생애주기 (App Data Lifecycle & Networking)

> 이 문서는 안드로이드 앱 내부에서 파싱된 콜 객체들이 탄생해서 서버로 넘어가기까지의 전체 생애 주기와 API 발송 단계를 설명하는 마스터 문서입니다. 현재 시스템에 100% 구현되어 작동 중입니다.

---

## 0단계: 서버 부팅 및 디폴트 필터 세팅 (Filter Initialization)
서버(Node.js)가 부팅되면 DB 테이블(`user_filters`)에서 디폴트 필터를 메모리(`baseFilter`)에 세팅합니다.
- 앱이 아직 한 번도 `/api/scrap`을 호출하지 않은 상태에서는 앱 내부에 `SharedPreferences`에 저장된 이전 세션의 필터가 남아있거나, 없으면 기본값(빈 필터)을 사용합니다.

## 1단계: 앱 부팅 및 감시 엔진 점화 (App Boot)
앱이 켜지면 `TelemetryManager.start()`가 호출되어 20초 하트비트 타이머를 세팅합니다.
접근성 서비스 화면 변경이 감지되면 `handleListScreen`이 호출되어 **파싱이 가장 먼저 실행**됩니다.

**📖 인성앱 파싱 문자열 예시:**
> `9.0 42.8 @ 경기 광주시 서울 강남구 1t 7.6`
*(위 콜은 차종 앵커 `1t`를 기준으로 오더 객체로 탄생합니다.)*

## 2단계: 콜 파생 및 1차 검문 (Parsing & Deduplication)
화면에 리스트가 나타나면 제일 먼저 차종(`1t`, `오` 등)을 기준점으로 위아래 글자들을 묶어 파싱합니다.
- `pickup`: "경기 광주시"
- `dropoff`: "서울 강남구"
- `fare`: 1t 옆의 7.6 → 76,000원
- `pickupDistance`: 9.0km
- `vehicleType`: "1t"

**검문 (중복 제거):**
- **기준:** 파싱 후 `(출발지 + 도착지 + 요금)` 문자열을 합쳐 해시(Hash)를 만듭니다.
- **캐싱 로직:** 이전에 본 해시값이면 서버 부하 방지를 위해 버리고, 처음 본 값이면 `processedOrderHashes`에 등록 후 다음 단계로 넘어갑니다. (최대 100개 유지)

## 3단계: 스크랩 발송 및 필터 수신 (`POST /api/scrap`)
파싱된 콜이 `enqueue()`로 `scrapBuffer`에 들어가면, 300ms 디바운스 대기 후 `flush()`가 호출되어 **`POST /api/scrap`이 호출**됩니다.
- **응답:** 서버는 즉시 반환하며, 응답 꼬리(Piggyback)로 최신 필터 정보를 한꺼번에 내려줍니다.
- 앱은 `dispatchEngineArgs`(=필터 정보)를 받자마자 `SharedPreferences`의 `"activeFilter"` 키에 덮어씌웁니다.
- **생존신고:** 화면이 20초간 멈춰있어도 빈 배열(`[]`)로 생존 핑을 보내며 최신 필터를 동기화합니다.

## 4단계: 서버 태세 역명령 하달 (Feedback Loop)
3단계 응답을 받을 때, 서버가 `deviceControl.mode` 값으로 근무 모드 명령을 내립니다.
- `"MANUAL"`: 수동 모드 유지 (관제 전용 - 클릭하지 않음)
- `"AUTO"`: 자동 사냥 모드 돌입 (아래 5단계 필터 가동)
- `"SHUTDOWN"`: 퇴근 명령 → 앱은 즉시 감시를 중단합니다.

---

*(이하 5~7단계는 4단계에서 `AUTO` 모드를 수신한 이후에만 실행되는 사냥 파이프라인입니다.)*

## 5단계: 자동 사냥(AUTO) 5대 조건 판별 (Smart Filter)
`AUTO` 모드 상태에서 새로운 콜이 파싱되면, 앱은 서버로 보내기 전에 이 콜이 '꿀콜'인지 **자체 검증**합니다.
1. **차종:** 파싱된 `vehicleType`이 허용 차종 배열에 포함되어 있는가?
2. **도착지:** `dropoff` 텍스트가 타겟 지역(destinationKeywords)에 있는가?
3. **요금 하한:** 76,000원이 최소요금(`minFare`) 이상인가?
4. **상차지 거리:** 접근거리 9.0km가 필터 반경(`pickupRadiusKm`) 이하인가? (합짐일 땐 바이패스)
5. **블랙리스트:** 텍스트 내에 기피 단어(`excludedKeywords`)가 없는가?

✅ **결과:** 모두 통과하면 터치(광클) 액션으로 넘어갑니다. 통과하지 못하면 3단계(스크랩)로 좌천됩니다.

## 6단계: 콜 1차 선점 및 낚아채기 보고 (`POST /api/orders/confirm`)
필터를 통과해 꿀콜을 광클하는 데 성공했다면, 앱은 사냥 집중 상태가 됩니다.
- **API 호출:** 화면에서 구한 요약본을 즉시 관제탑에 보고합니다. (`step: BASIC`)
- **응답 대기:** 서버는 이를 이용해 관제탑 화면에 '진행 중' 상태를 띄워줍니다.

## 7단계: 상세 정보 수집 후 서버 최종 판결 대기 (`POST /api/orders/detail`)
선점에 성공한 앱은 상세 팝업창 3개(적요, 출발, 도착)를 순식간에 열었다 닫으며 장문 텍스트를 수집합니다.
- **API 호출:** 완성된 상세 데이터를 서버로 넘기고, 영구히 유지할지(KEEP), 뱉을지(CANCEL) '최종 결재'를 서버에 요청합니다.
- **롱폴링 대기 (Death Valley):** 서버에서 카카오 맵 API를 호출해 동선과 요율을 계산하고, 사장님이 승인/거절을 클릭할 때까지 **최대 30초를 살얼음판처럼 대기**합니다.
- **서버 Response:** `{"action": "KEEP" | "CANCEL"}` 명령이 떨어지면 응답에 맞게 화면의 닫기/취소 버튼을 클릭하고 임무를 완수합니다.

## 📝 주요 용어 주석 (Glossary)

* **`scrapBuffer`**: `TelemetryManager` 안에 있는 메모리 리스트. 파싱된 콜 객체를 서버로 보내기 전에 잠깐 담아두는 대기실.
* **`300ms 디바운스`**: 화면이 깜빡일 때 연달아 들어오는 이벤트를 모아서 300ms 동안 잠잠해질 때 한 방에 묶어 전송하는 부하 방지 기법.
* **`dispatchEngineArgs / activeFilter`**: 서버가 `/api/scrap` 응답에 피기백으로 내려주는 필터 객체. 앱은 이 값을 SharedPreferences에 저장하고 광클 조건 검사에 사용.
