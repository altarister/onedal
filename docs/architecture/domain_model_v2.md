# 🏗️ [1DAL 도메인 모델 재설계] 시니어 평가 및 제안

> **작성일**: 2026-05-04  
> **목적**: 기존 `loadState` 짬뽕 설계를 3계층 도메인 모델로 분리하여 하드코딩 버그를 원천 차단하고, 확장 가능한 아키텍처를 확보한다.

## 📋 업계 조사 결과 (24시화물 / 원콜 / 인성 비교)

주요 화물 앱에서 모두 상차/하차 시점에 명시적 확인(서명/사진/버튼) 절차가 존재합니다:

| 앱 | 배차 확정 | 상차 완료 | 하차 완료 | 서명/증빙 |
| :--- | :--- | :--- | :--- | :--- |
| **인성** | 콜 확정 버튼 → 사무실에 자동 통보 | 상차 완료 터치 (픽업 서명) | 하차 완료 터치 (수취인 서명) | ✅ 전자 서명 |
| **24시화물** | 화물배정 버튼 → 배차 확정 | 상차완료 버튼 | 하차완료 + 전자인수증 | ✅ 전자인수증 + 사진 |
| **원콜** | 배차 수락 → 배정 | 상차 완료 처리 | 운송 완료 처리 | ✅ 카메라 촬영 증빙 |

**결론**: 업계 표준으로 `배차확정 → 상차완료 → 하차완료`라는 **3단계 전환**이 존재하며, 각 전환 시점에 서명이나 사진 같은 **명시적 증빙(POD: Proof of Delivery)**이 붙습니다.

---

## 🔍 현재 설계의 근본 문제

핵심 문제: **"사용자 상태", "차량/짐 상태", "시스템 전략"이 하나의 변수에 짬뽕되어 있다.**

현재 코드의 `loadState`는 사실 3가지 서로 다른 관심사(Concern)를 하나로 뭉뚱그리고 있습니다:

1. **기사님이 지금 뭘 하고 있는가?** (대기? 운전? 상차? 하차? 밥?) → 사용자 행동
2. **내 콜은 지금 어떤 단계인가?** (심사중? 확정? 상차완료? 하차완료?) → 오더 생명주기
3. **시스템은 어떤 반경으로 콜을 찾아야 하는가?** (첫짐? 합짐? 경로상?) → 사냥 전략

이 3가지를 분리하지 않으면 영원히 맥락을 잃고 하드코딩 버그가 반복됩니다.

## 💡 제안: 3계층 도메인 모델

### 계층 1. 기사 행동 상태 (`DriverAction`)
> "기사님이 물리적으로 지금 뭘 하고 있는가?"

이것은 **기사님이 직접 버튼을 눌러서** 바꾸는 상태입니다. 시스템이 멋대로 바꾸면 안 됩니다.

```typescript
type DriverAction = 
  | 'WAITING'     // 콜 대기중 (주차 상태에서 앱 보며 콜 고르는 중)
  | 'DRIVING'     // 운전중 (상차지든 하차지든 어딘가로 이동 중)
  | 'LOADING'     // 상차중 (픽업지에서 물건 싣는 중)
  | 'UNLOADING'   // 하차중 (하차지에서 물건 내리는 중)
  | 'RESTING'     // 휴식중 (밥, 화장실, 일시정지)
```

**포인트**: 여기서 `LOADING`과 `DRIVING`은 말 그대로 기사님의 **물리적 행위**를 뜻합니다. 기존 코드의 `loadState: 'LOADING'`(=확정콜 있는 상태)과는 완전히 다른 개념입니다. 이름이 같아도 의미가 다르기 때문에, 기존 `loadState`를 완전히 폐기하고 새 이름으로 가야 헷갈리지 않습니다.

### 계층 2. 오더 생명주기 (`OrderLifecycle`)
> "이 콜(퀵 오더)은 지금 어떤 단계인가?"

**확정 전(심사 중)**과 **확정 후(내 퀵)**는 완전히 다른 타입으로 분리해야 합니다.

#### A. 심사 중 오더 (`PendingOrder`) — 서버 메모리 전용
앱이 긁어와서 서버가 꿀/똥콜을 판별 중인 임시 데이터. 기사가 KEEP/CANCEL 누르기 전.
```typescript
interface PendingOrder extends OfficeOrder {
  phase: 'SCREENING' | 'AWAITING_DECISION';  // 서버 연산 중 | 관제탑 결재 대기
  capturedDeviceId: string;
  capturedAt: string;
  // ... 카카오 연산 결과, 꿀/똥콜 판정 등
}
```

#### B. 확정 오더 (`MyOrder`) — 내 소유의 퀵 오더
기사가 확정(KEEP)해서 **내가 책임지고 수행해야 하는 퀵**. 업계 표준 3단계를 그대로 반영:
```typescript
type MyOrderStatus = 
  | 'CONFIRMED'   // 배차 확정 (아직 상차지로 안 감)
  | 'PICKED_UP'   // 상차 완료 (물건 실음, 서명/사진 촬영)
  | 'DELIVERED'   // 하차 완료 (물건 내림, 수취인 서명)
```

**`canceled`를 뺀 이유**: "내 확정 퀵"에 `canceled`가 들어가는 건 말이 안 됩니다. 취소된 오더는 `MyOrder` 배열에서 아예 빠지고, 별도의 취소 이력 로그로 관리합니다.

### 계층 3. 사냥 전략 (`DispatchPhase`) — 파생(Derived) 데이터
> "시스템이 어떤 반경으로 콜을 찾아야 하는가?"

이 값은 **DB에 저장하거나 하드코딩하지 않습니다**. 계층1(기사 행동)과 계층2(확정 콜 수)를 조합하면 자동으로 계산됩니다.

| 확정 오더 수 | 기사 행동 | → 사냥 전략 (DispatchPhase) | 우회 반경 |
| :--- | :--- | :--- | :--- |
| 0개 | `WAITING` | **`STANDBY`** (첫짐 탐색) | DB설정값 기반 원형 반경 |
| 1개+ | `WAITING`, `LOADING`, `UNLOADING` | **`GATHERING`** (합짐 탐색) | DB→Base→Active 필터의 `corridorRadiusKm` (예: 5km) |
| 1개+ | `DRIVING` | **`DELIVERING`** (경로상 탐색) | 시스템 강제 0km (DB 원본값 건드리지 않음) |

```typescript
// 이 함수는 순수 계산 함수 (Pure Function) — DB 접근/하드코딩 없음
function deriveDispatchPhase(
  driverAction: DriverAction, 
  confirmedOrders: MyOrder[]
): DispatchPhase {
  if (confirmedOrders.length === 0) return 'STANDBY';
  if (driverAction === 'DRIVING') return 'DELIVERING';
  return 'GATHERING';
}
```

---

## ✅ 이 설계의 장점

1. **하드코딩 원천 차단**: `corridorRadiusKm`를 0이나 10으로 강제 덮어쓰는 코드가 완전히 사라집니다. 반경은 항상 기사님의 설정값(DB→Base→Active 경로)에서 가져오고, 주행 중일 때만 엔진이 계산 시점에 0으로 취급합니다.

2. **확장성**: "차에 짐이 얼마나 실렸는지", "서명/사진 증빙" 같은 기능을 나중에 추가할 때, 이 3계층 모델 위에 깔끔하게 붙일 수 있습니다. (예: `MyOrder`에 `pickupSignature`, `deliveryPhoto` 필드 추가)

3. **단일 진실 원천(Single Source of Truth)**: 기존에는 `loadState`를 바꾸면 반경이 바뀌고, 반경을 바꾸면 상태가 바뀌는 순환 참조가 있었습니다. 이제는 기사 행동과 오더 수가 진실이고, 사냥 전략은 그로부터 파생됩니다.
