# 1DAL 글로벌 기획서 (Global Master PRD)

본 문서는 `onedal-app`(전방 스캐너)과 `onedal-web`(관제탑 서버) 두 개의 애플리케이션이 융합되어, 물류 현장의 악질 생태계와 속도전을 어떻게 극복하고 궁극적인 AI 배차 관제 시스템으로 거듭나는지를 정의한 핵심 설계도입니다.

---

## 🕒 1. 기기별 폴링 & 선점 경합 타임라인 (3-Phones 시나리오)
인성앱의 고유 기능(동기화)과 매크로들의 경합 상황을 시간대별로 재구성한 기초 원리입니다.

| :--- | :--- | :--- | :--- | :--- |

| **T+6초** | 알럿 확인 후 리스트로 쫓겨남 | `[닫기]` or `[취소]` 선택 | 다음 로딩 대기 중 | 2번 폰 연결 유지 중 |

*   **분석 포인트**: 2번 폰이 늦게 리스트를 받았음에도, 선빵(광클)을 통해 인성 서버의 권한을 획득했습니다. 확정 후에도 우측 상단 박스 위에 활성화된 `[취소]` 버튼을 통해 우리는 최악의 똥콜을 버릴 **[데스밸리 30초]** 를 확보할 수 있습니다.

## ⚡ 2. 100% 서버 하이브리드 연산 모델 (Shoot First, Ask Later)

### 🛡️ [1단계: 무지성 엣지 광클 (0.01초 기계적 선빵)]
폰은 선택하지 않은 리스트(쓰레기 콜) 데이터도 서버로 보내 필터 업데이트용으로 활용합니다. 폰은 로컬(로딩된 폰 메모리)에 저장된 관제탑 필터 조건만 맞으면 **무조건 `[확정]`** 을 갈겨 콜을 훔쳐 옵니다.

### 🧠 [2단계: 관제탑 30초 정밀 검수 (데스밸리 타임)]
콜을 삼킨 폰은 화면의 텍스트를 `onedal-web` 서버로 던지고 연산 결과 리턴을 기다립니다.
*   **팝업 레이스 컨디션 방어**: DOM 렌더링 픽셀 잔상을 감지해 터치 누락을 방지하며 가장 정확한 출발지/도착지 텍스트를 0초 컷 스크랩.
*   **능동형 Data 파서**: `고객`, `위치` 등 라벨과 값이 개행 분리된 최신 UI에 맞춰 Multiline 인텔리전스 파싱 동작.
*   **3중 구조 카카오 지오코딩 (Fallback)**: 카카오 로컬 API 거절 시, `괄호 도려내기 ➡️ 장소/키워드 양방향 검색 ➡️ 4어절 절사`를 통해 지저분한 주소도 무조건 좌표 변환 강행.
*   **Fail-Safe 장치**: 서버 무응답 시 무조건 **'30초 후 스스로 앱 내 [취소] 클릭'** 타이머 가동.
*   **서버 API 리턴**: 카카오 API 통행료/시간 계산 후 `{"action": "확정(유지)"}` 또는 `{"action": "취소"}` 정보 응답.

### 🗑️ [3단계: 데스밸리 판정에 따른 폰 동작 (뱉기 vs 사냥 재개)]
*   **똥콜 판정 (취소 하달)**: 툴비/까대기로 취소 명령을 받으면 즉결 상단의 `[취소]` 버튼 터치 (페널티 감수).
*   **꿀콜 판정 (사냥 재개)**: 유지 명령을 받으면 즉시 하단 **`[닫기]` 버튼 터치하여 리스트 화면으로 복귀 (다음 합짐 사냥)**.

### 🗼 [4단계: 관제탑(서버)이 해야 할 핵심 후속 작업]
콜 배차가 무사히 확정 유지되었다면 관제탑은 백그라운드 작업을 실행합니다:
1.  **필터 값 전파**: 필드에 나간 모든 폰들에게 **추가/변경된 합짐 타겟 필터(회랑 좌표)** 를 업데이트(Push).
2.  **카카오 내비 전송**: 경유지가 추가된 스마트 TSP 정렬 카카오맵 링크를 기사님 카카오톡으로 전송.
3.  **운행 일지 DB 세팅**: DB에 오더 row를 생성하여 즉시 통화 내용(화물 제원)을 입력할 수 있게 대비.

## 🎛️ 3. 기기 원격 관제 및 텔레메트리 (Remote Fleet Control)
관제 웹에서 개별 폰의 사냥 성과를 보고 `[오토-확정]`, `[반자동-읽기만]` 토글을 원격 제어하여 누적 페널티 킬스위치(Kill-Switch) 관리.

---

## 🏧 4. 출근 전 프리플라이트 (잔고 락 & 엇박자 부팅)
*   **엇박자 폴링(Time-Staggered Polling)**: 기기 3대를 1.5초 시차를 두고 켜서, 인성 서버의 5초 주기 새로고침의 한계를 우회하여 **1.6초마다 실시간 폴링**을 거두는 초단타 체계 구축.

---

## 🚫 5. 바닥콜(Floor Call) 지뢰 탐지기 (Toxic Filter)
2분 넘게 살아있는 콜은 통계적 똥콜(지뢰)로 간주. 광역주소 암막 화주, "전화 금지" 등 갑질 적요는 폰에 저장된 필터 선에서 0.001초 만에 스킵.

---

## 🧭 6. 동적 합짐(On-The-Way) 레이더 및 카카오 TSP 통합 라우팅
*   **지리 회랑(Corridor) 필터링**: 현재 배차된 본콜의 이동 방향(가상의 5km 파이프라인) 반경 안에 떨어지는 콜만 앱폰의 레이더망에서 1차 여과.
*   **초정밀 노선 기반 합짐 상차 필터링 (스마트 회랑 추출) 🌟 NEW 🌟**: 첫 콜(본콜) 확정 시 카카오 경로 API 응답 데이터에 포함된 '모든 세부 도로명'을 추출합니다. 이 도로명들과 면적/인접성이 겹치는 읍/면/법정동 단위 지역명을 맵핑하여 데이터베이스화하고, 도출된 지역명 리스트를 다음 합짐 콜 사냥(앱폰 스캐너)의 상차지 필터로 즉각 주입합니다. 이를 통해 이동 노선에서 한 블록도 크게 벗어나지 않는 극강의 효율적 합짐을 구성합니다.
*   **서버-투-서버(S2S) 우회 연산**: 인성앱에서 낚아챈 합짐 콜의 데이터를 **관제탑 백엔드(Node.js RAM)**가 직접 물고 카카오 모빌리티 API와 즉각 통신. 프론트엔드의 부하를 없애고 0.2초 내로 기존 노선 대비 '추가 소요시간(분)'과 '추가 거리(km)' 패널티를 정밀 도출.
*   **통합 타임라인 시각화 (Subway UI)**: 복잡하게 얽힌 합짐(경유지)들의 방문 순서를 관제 대시보드 상단에 지하철 노선도 형태(`상차 ➡️ 상차 ➡️ 하차 ➡️ 하차`)로 일직선화 배열. 운전 중 직관적인 가시성 제공.

아침에 일 시작 할때 내 디폴트 설정을 가지고 와서 출발지에서 2콜을 잡고 이동하기 시작해 
그럼 gps 좌표가 바뀌면서 필터가 자동 축소 되어야 하는거지 그래서 가는동안 같은 경로에 있는 콜을 잡아 오전 3콜을 마무리 한다. 점심 이후 다시 도착 희망 시/도를 내가 살고 있는곳으로 바꾸고 2콜을 잡아 출발 하고 필터가 바뀌면서 오는도중 같은경로 로 콜을 잡아 오후 3콜로 마무리 한다. 
이것이 내 시나리오야.

첫콜은 도착 희망 시/도를 설정하고 모든 짐을 다잡는걸 목표로 금액이 높은 짐을 우선으로 잡는거야. 
첫짐을 잡았다면 트럭에 남아 있는 공간도 생각해서 그 첫 콜의 경로에 설정된반경(10km) 까지 돌아가더라도 콜을 잡고 알려주면 내가 판단해서 킵과 켄슬을 결정할께
그래서 2번쩨 콜을 잡았다면 합짐한 경로로 배달을 위해 난 출발 할꺼야 운전중이니 판단을 최소화 하기위해 설정된 반경을(0km)로 바꾸어 출발지가 계속 바뀌는 가는길에 있는콜만 잡는거지. 

복귀콜을 시작할때는 0원짜리 우리집을 넣을수 있어야 하겠다.. 아니면 도착 희망 시/도를 바꿔야 하는데.. 그럼면 아침마다 복귀콜 할때 마다 필터설정을 계속 바뀌어야 하니까.. 
상차지 하차지 도착은 관제 엡에서 해야 할까? 아니면 경로를 추적해서 목적지 근처에 오면 자동으로 지워 줄까. 이것도 기획을 해야 할듯
