## 1. 테스트 계층 구조

```
┌──────────────────────────────────┐
│    E2E (수동 검증)               │  ← 실제 기기 + 인성앱으로 테스트
├──────────────────────────────────┤
│    통합 테스트                    │  ← 서버 Mock + 앱 내부 모듈 결합
├──────────────────────────────────┤
│    단위 테스트                    │  ← Parser, Filter 로직 독립 검증
└──────────────────────────────────┘
```

## 2. 단위 테스트 (Unit Test)

순수 로직만 테스트합니다. 안드로이드 프레임워크(AccessibilityService, SharedPrefs 등)에 의존하지 않는 코드만 대상입니다.

### 2-1. ScrapParser 파싱 테스트

| 테스트 케이스 | 입력 (screenTexts) | 기대 출력 |
|-------------|-------------------|----------|
| 정상 인성콜 리스트 | `["신규", "빠른설정", "오", "@역삼동", "/강남구", "요금", "55,000"]` | `pickup="역삼동"`, `dropoff="강남구"`, `fare=55000` |
| 요금 없음 | `["신규", "@미상", "/미상"]` | `fare=0` |
| 빈 배열 | `[]` | `pickup="미상"`, `dropoff="미상"`, `fare=0` |
| 특수문자 포함 | `["@역삼 (강남)", "/LG로지스"]` | `pickup="역삼 (강남)"`, `dropoff="LG로지스"` |

### 2-2. shouldClick 필터 테스트

| 테스트 케이스 | FilterConfig | Order | 기대 결과 |
|-------------|-------------|-------|:---:|
| 운임 합격 | `minFare=30000` | `fare=55000` | ✅ click |
| 운임 불합격 | `minFare=30000` | `fare=10000` | ❌ skip |
| 차종 필터 | `allowedVehicleTypes=["다","라"]` | `vehicleType="오"` | ❌ skip |
| 제외 키워드 | `excludedKeywords=["제주"]` | `dropoff="제주시"` | ❌ skip |
| 빈 필터 (전체 허용) | `allowedVehicleTypes=[]` | 아무거나 | ✅ click |

### 2-3. ScreenContext 판별 테스트

| 입력 텍스트 | 기대 상태 |
|-----------|----------|
| `"신규 빠른설정 오 @역삼 /강남 요금 55000"` | `LIST` |
| `"적요상세 요금 확정 배차"` | `DETAIL_PRE_CONFIRM` |
| `"적요상세 요금"` (확정 없음) | `DETAIL_CONFIRMED` |
| `"출발지 상세 전화1"` | `POPUP_PICKUP` |
| `"취소할 수 없습니다"` | `POPUP_ERROR` |
| `"카카오톡 채팅"` | `UNKNOWN` |

### 2-4. 동명이동 검증 테스트

| 하차지 | customCityFilters | 화면 텍스트 | 기대 결과 |
|-------|-------------------|-----------|----------|
| `"신사동"` | `["서울 강남"]` | `"강남구 신사동"` | ✅ 2단계 통과 |
| `"신사동"` | `["서울 강남"]` | `"인천 남동구 신사동"` | ❌ 3단계 CANCEL |
| `"역삼동"` | 아무거나 | 아무거나 | ✅ 일반 콜 (CAUTION_DONGS 아님) |

## 3. 통합 테스트 (Integration Test)

### 3-1. API Mock 서버 연동

로컬에 Mock 서버를 띄우고 앱의 전체 통신 흐름을 검증합니다.

| 시나리오 | Mock 서버 동작 | 앱 기대 행동 |
|---------|--------------|------------|
| 정상 Piggyback KEEP | `/scrap` 응답에 `decision: {action: "KEEP"}` | 데스밸리 해제, 판결 콜백 발동 |
| Piggyback CANCEL | `/scrap` 응답에 `decision: {action: "CANCEL"}` | "취소" 버튼 자동 클릭 |
| 서버 무응답 | `/detail` 타임아웃 | `CANCEL`로 간주, 비상 보고 |
| ACK 프로토콜 | 판결 수신 후 다음 `/scrap`에 `ackDecisionId` 포함 확인 | ACK 전송 후 `pendingAckDecisionId` 삭제 |

### 3-2. EngineRouter 플러그인 스위칭 테스트

| 시나리오 | packageName | 기대 동작 |
|---------|------------|----------|
| 인성콜 감지 | `"com.insung.app"` | InsungParser + InsungAutomationEngine 라우팅 |
| 24시 감지 | `"com.cargo24.app"` | Cargo24Parser 라우팅 (TODO) |
| 카카오톡 감지 | `"com.kakao.talk"` | `null` 반환 → 스캔 중단 |
| 비활성화된 앱 | 인성콜이지만 enabledApps에 없음 | `null` 반환 |

## 4. 수동 E2E 검증 체크리스트

실제 기기 + 실제 인성앱으로 검증해야 하는 항목들입니다.

### 🟢 기본 플로우

- [ ] 앱 최초 설치 → PIN 입력 → 기기 등록 성공
- [ ] 접근성 ON → 서버 온라인 표시 확인
- [ ] 인성앱 이동 → 콜 리스트 스캔 시작 확인 (미니 로그에 표시)
- [ ] AUTO 모드에서 꿀콜 자동 클릭 확인
- [ ] 확정 후 팝업 서핑 (적요→출발지→도착지) 순서 확인
- [ ] 서버 판결(KEEP) 수신 후 정상 복귀 확인
- [ ] 접근성 OFF → 서버 오프라인 통보 확인

### 🟡 엣지케이스

- [ ] 동명이동 의심 동네 → 3단계 검증 → 정상 판단
- [ ] 데스밸리 30초 타임아웃 → 자동취소 + 비상 보고
- [ ] "취소할 수 없습니다" 팝업 → POPUP_ERROR 감지 + 비상 보고
- [ ] 기사님 수동 뒤로가기 → 세션 초기화 정상 동작
- [ ] 화면 꺼짐 + 다시 켜짐 → 텔레메트리 재개 확인

### 🔴 위험 시나리오

- [ ] AUTO 사냥 중 인성앱 강제 종료 → 엔진 정상 대기 모드 복귀
- [ ] 네트워크 끊김 5분 → 재연결 후 텔레메트리 정상 재개
- [ ] 장시간(8시간) 연속 가동 → 메모리 누수 없음 (해시 캐시 정리 동작 확인)

## 5. 리팩토링 전후 회귀 테스트

리팩토링 시 **절대 깨지면 안 되는** 핵심 동작 체크리스트:

| # | 동작 | 검증 방법 |
|---|------|----------|
| 1 | 리스트 파싱 정확도 | 동일 화면 텍스트 → 동일 출력 비교 |
| 2 | 광클 좌표 정확도 | 실기기에서 올바른 콜이 클릭되는지 |
| 3 | 팝업 서핑 순서 | 적요→출발지→도착지 순서 유지 |
| 4 | Piggyback ACK | 판결 수신 → ACK → 판결 삭제 흐름 |
| 5 | 데스밸리 타이머 | 정확히 설정 시간(30/40/50초) 후 발동 |
| 6 | 고스트 방어 | 과거 orderId 판결 폐기 |
| 7 | 팝업 잔상 방어 | 닫기 직후 오탐 없음 |
